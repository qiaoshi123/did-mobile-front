## 首次使用 create-proposal 生成 Spec + 问题修复

使用 create-proposal Skill 为登录页生成了 Spec 实施计划：`docs/specs/login-spec.md`

### 发现的三个问题及修复

用户反馈 Spec 中登录逻辑代码存在三个问题：

1. **未使用已有的 `useLoading` Hook**：自行写了 `loading.value = true/false` + `try/finally`，而项目已有 `useLoading` 提供 `withLoading` 方法
2. **未遵循 HTTP 响应约定**：写了 `try/catch` 模式，但项目的 API 封装不会抛业务异常，应该用 `res.ok` 判断成功/失败，失败时 `uni.showToast({ title: res.msg })`
3. **自行推断创建了 Store**：用户没说要创建 `useUserStore`，但 AI 根据"常识"推断"登录后肯定要存 token"就自行生成了创建任务

### Skill 修复（create-proposal SKILL.md）

修改了 4 处：

1. **核心原则**新增 3 条：
   - 第 3 条增强：Spec 代码必须使用已有 Hooks/Utils，禁止自行实现替代方案
   - 第 8 条（新）：遵循 HTTP 响应约定，禁止自行发明错误处理模式
   - 第 9 条（新）：不虚构交互逻辑，用户没明确的行为标注"⚠️ 待确认"

2. **第 0 步扫描**新增 2 项：
   - 第 4 项增强：读取每个 Hook 的源文件（了解返回值和用法）
   - 第 9 项（新）：读取 HTTP 响应约定（`CoreResponse` 结构 + 请求封装的错误处理模式）

3. **第 1 步**强化 Store 和 Hooks 的约束：
   - Store：禁止根据"常识"推断，如果 AI 认为"很可能"需要 Store 应标注"⚠️ 待确认"
   - Hooks：必须强制使用已有 Hook（loading→useLoading、分页→usePagination、表单→useFormValidation）

4. **检查清单**新增 4 项检查

之前错误地手动修改了 `login-spec.md`。正确做法是：**Spec 是 Skill 的产物，产物有问题应修好 Skill/Rule 后重新生成，而非手动修补。** 已删除旧 Spec（`docs/specs/login-spec.md`），后续需用修好的 Skill 重新生成验证。

**正确流程**：发现 Spec 问题 → 定位 Skill/Rule 缺陷 → 修复 Skill/Rule → 删除旧 Spec → 用修好的 Skill 重新生成

create-proposal Skill 的核心原则第 8 条和第 0 步第 9 项之前写的是"去扫描 `src/http/core/types.ts` 和 `request.ts` 源码来了解 HTTP 响应约定"——这是多余的，因为 `05-API规范` Rule 的「业务层调用约束」已经明确写了 `res.ok` 判断成功、`res.msg` 可直接展示。

修复：

## 新增 create-proposal Skill

创建了 `create-proposal` Skill（`.codebuddy/skills/create-proposal/`），方案四三层架构中的"大脑"层。

核心职责：将 UI 清单 + API 文档 + 交互说明 → 拆解为有序的 Spec 实施计划，每个任务对应一次 create-xxx Skill 调用。

文件结构：
- `SKILL.md` — 主文件（5 步工作流：扫描现有资源 → 需求分析 → 任务拆解 → 编写任务详情 → 输出 Spec 文档）

核心原则：
- 一个任务 = 一次 Skill 调用
- 执行指令自给自足（AI 无需翻阅原始文档）
- 已有资源不生成创建任务
- 执行顺序：API → Store → Hook/Util → 组件 → 页面
- 缺失信息标注"待补充"，不虚构

## create-proposal Skill 重大升级 v3：API 使用清单强制化 + Spec 唯一真相源

### 1. API 使用清单从"可选"升级为"必填"

## create-proposal Skill 重大重写 v4：回归流程控制本质

用户反馈 Spec 存在两个严重问题：
1. **越权**：Spec 里塞了大量"教 AI 怎么做"的实现细节（Store 字段定义、Hook 函数签名、校验规则代码等），这些应该是 create-xxx Skill 的职责
2. **AI 自作主张**：确认阶段只问了 API 一个问题就直接输出了定稿，大量决策（拆 Store/Hook、字段设计、跳转方式等）都是 AI 自行决定的

用户明确了 Spec 的定位：**开发流程控制文档** — 保证需求不遗漏、开发不脱节、可复盘归档。

### SKILL.md 重写要点

1. **砍掉第 3 步（编写任务详情）中所有"执行指令必须包含的信息"** — 之前按 Skill 类型列出了 Store 要写什么字段、Hook 要写什么入参返回值等，全部删除
2. **确认机制全面加强** — 从只确认 API 改为必须确认：任务拆分方案 + 交互决策 + 所有关键决策点
3. **流程从 7 步精简为 6 步**（0-5）：资源扫描 → API 确认 → 需求分析+草稿确认（核心关卡）→ 输出定稿 → 交付确认 → 开始执行
4. **新增"AI 角色边界"** — AI 是流程编排者不是设计决策者，不猜不推断不发挥
5. **新增"任务详情写法原则"** — 用正确/错误示例对比，明确"需求描述"vs"实现指导"的边界
6. **核心分工表** — Spec 说"做什么"，Skill 决定"怎么做"，Rule 约束"做的对不对"
