## 新建三个 Skill：create-page、create-component、create-store

**设计原则**：Rule 管"标准"（始终生效），Skill 管"流程"（按需触发）。Skill 之间不互相调用，通过共享 Rule 实现一致性。

**create-page**（`.codebuddy/skills/create-page/SKILL.md`）：
- 引用 Rule：02-编码规范、03-项目结构、04-组件规范
- 核心：第 0 步"组件扫描与匹配"——读 components/index.ts + 查 tdesign-uniapp Skill → 输出组件方案表格 → 再写代码
- 写页面过程中如需新建组件，直接按 04-组件规范 Rule 创建，不需要调 create-component Skill

**create-component**（`.codebuddy/skills/create-component/SKILL.md`）：
- 引用 Rule：04-组件规范、02-编码规范
- 核心：确认组件定位（通用/页面级）→ 创建文件 → 编写代码 → 注册到 index.ts
- 支持场景：通用组件、页面级组件、二次封装 TDesign 组件
- 独立入口，给用户单独说"创建组件"时触发

**create-store**（`.codebuddy/skills/create-store/SKILL.md`）：
- 引用 Rule：07-状态管理、02-编码规范
- 核心：确认 Store 定位（模块名、职责、是否持久化）→ 创建文件 → Composition API 风格编写 → 注册到 index.ts
- 支持场景：新建 Store 模块、在已有 Store 中追加状态/方法

**04-组件规范 Rule 同步更新**：
- 组件选择优先级"信息获取方式"列措辞：从"加载 tdesign-uniapp Skill"改为"查阅 tdesign-uniapp Skill 中的组件文档"

## Skill 自查修复：消除 AI 决策盲区

**排查标准**：判断错的代价高 + AI 容易判断错 → 必须改；代价低或 AI 能准确判断 → 保持默认值不改。

**修改 1 — create-page 第 0 步扩大扫描范围**：
- 原来只扫 `src/components/index.ts` + TDesign → 现在增加扫 `src/pages/*/components/` 所有页面级组件
- 发现其他页面有同名/同功能组件时：禁止跨页面引用、禁止默默复制，必须询问用户是否提升为通用组件
- 优先级从 3 级扩展为 4 级：通用组件 > TDesign > 复用已有页面级组件（需用户确认提升） > 新建

**修改 2 — create-store 模板 persist 矛盾**：
- 文字规则说"默认不开启 persist"，但模板代码里白纸黑字写着 `persist: true` → 自相矛盾
- 改为模板中不含 persist 选项，下方注释说明"用户要求时才加"

**修改 3 — create-component 加重名检查**：
- 确认组件名称后扫描 `src/components/` 和 `src/pages/*/components/` 是否已有同名组件
- 通用组件已有同名 → 询问用户是修改现有还是换名新建
- 其他页面已有同名 → 询问用户是否提升为通用组件

**修改 4 — create-store 加重名检查**：
- 确认模块名后检查 `src/store/<module-name>.ts` 是否已存在
- 已存在 → 询问用户是修改现有（转场景二）还是换名新建

**修改 5 — create-page 加页面重名检查**：
- 注册路由前检查 `src/pages/<page-name>/` 目录和 `pages.json` 是否已有同名页面/路由
- 已存在 → 询问用户是修改现有页面还是换名新建

**未改的项（分析后认为不需要改）**：
- 组件放通用还是页面级：默认放页面级是合理低成本默认值，放错了移文件成本极低
- types.ts 是否拆分：AI 数字段数是机械判断，错了也不影响运行
