## create-page Skill 新增「场景二：修改已有页面」

### 设计决策
- **不新建独立 Skill**，而是在 `create-page` 中新增场景二分支（与 `create-store` 的「场景一/场景二」模式一致）
- 原因：两者共享完全相同的 Rules（03/04/06/08/09）和文档查阅工具，分两个 Skill 会制造触发词歧义

### 变更内容
1. **description 更新**：新增修改类触发词（修改页面、改页面、迭代页面、给页面加功能、调整页面、优化页面、页面改版、重构页面、页面对接接口）
2. **场景判断规则**：新建/创建/铺 → 场景一；修改/改/迭代/加功能 → 场景二；"写"但已存在 → 询问用户
3. **场景二 SOP 四步流程**：
   - 第 0 步（上下文全量读取）：读取页面 + 页面级组件完整代码，输出依赖图（组件/Store/API/工具函数/生命周期/路由参数）
   - 第 1 步（变更方案设计）：输出变更方案表格（文件 × 操作 × 变更内容），约束最小改动原则
   - 第 2 步（组件扫描）：如需新组件才执行，复用场景一第 0 步
   - 第 3 步（增量修改）：强制 `replace_in_file` 精准替换，禁止 `write_to_file` 重写；新增代码插入对应分区末尾
   - 第 4 步（影响检查）：检查 Props/Emits、Store、API、样式隔离、路由参数的兼容性，输出检查结果表格
4. **场景二独立检查清单**：14 项，覆盖上下文读取、依赖图、变更方案、增量修改、影响检查
5. **场景一原有内容完整保留**，仅调整为「场景一」标题层级，重名检查增加「用户确认修改 → 转场景二」的跳转逻辑
