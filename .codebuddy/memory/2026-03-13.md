## H5 跨域解决方案实施（方案 A：抽 JSON）

### 问题
H5 本地开发环境出现跨域请求，因为 client 中 baseUrl 使用了绝对路径 `http://ip:port`。

### 方案
三层架构：条件编译 + Vite 代理 + Nginx 反代

### 改动文件

1. **新建 `src/config/gateway.json`** — 网关配置唯一数据源（JSON），包含 runtimeEnv、gatewayName、gatewayConfig（四个环境：dev/preview/production/sywl）
2. **改写 `src/config/index.ts`** — 从 gateway.json 读取并导出，增加了 `GatewayNameKey`、`RuntimeEnvKey` 类型导出
3. **改造 4 个 client**（did/did-app/invoice/tdh）：
   - H5 环境：`#ifdef H5` 使用相对路径 `/${GATEWAY_NAME.xxx}/...`
   - 非 H5 环境：`#ifndef H5` 使用 `GATEWAY_CONFIG[runtimeEnv][GATEWAY_NAME.xxx]` 动态环境
4. **配置 `vite.config.ts`** — 从 gateway.json 动态生成 proxy 规则，根据 runtimeEnv 自动选择目标环境

### 切换环境方式
只需修改 `gateway.json` 中的 `runtimeEnv` 字段即可切换所有环境（dev/preview/production/sywl）。

## 优化 create-api Skill 的触发 description

### 问题
用户说"编写一个 API 接口"并提供接口文档时，Skill 没有被触发。原因是 `description` 过于笼统（只写了"创建 API 接口封装"），缺少用户常见表述的触发关键词。

### 改动
将 `SKILL.md` frontmatter 的 `description` 从一行改为详细的多行描述，使用 YAML `>` 折叠块语法，包含：
- 多种动词组合：创建、编写、新增、封装、添加、对接、接入
- 常见场景短语：根据文档写API、根据接口文档编写、按照API文档实现
- 触发关键词列表：API、api、接口、interface、endpoint、接口文档、API文档、swagger、请求封装等
- 兜底描述：只要涉及 HTTP 请求的接口封装工作，都应该触发此技能

## 修改 create-api Skill 和 05-API规范 Rule：新增公共参数识别流程

### 问题
接入「查询app版本」API 时，接口文档中有 `equipmentType` 请求头参数，但该参数已由 `requestCommonHeaderInterceptor` 拦截器自动注入。由于 skill 中没有"先检查拦截器已注入哪些公共参数"的步骤，导致 AI 在 API 函数中重复传入了 `equipmentType`。

### 改动思路
不硬编码具体字段名，而是定义通用流程：在编写 API 前，先读取拦截器源码，动态感知已自动注入的参数。

### SKILL.md 改动
1. 场景一步骤总览新增「第 0 步：识别公共参数」（必须先做）
2. 新增详细模板「第 0 步」：读取 client.ts → 找到请求拦截器 → 读拦截器源码 → 记录自动注入的字段 → 后续步骤排除这些字段
3. 检查清单新增 2 项：已执行第 0 步、公共参数未被重复写入

### 05-API规范.md 改动
1. 禁止事项新增第 11 条：禁止在 API 函数中重复传入拦截器已自动注入的公共参数
2. 新增 API 检查清单新增 2 项：已读取拦截器源码识别公共参数、公共参数未被重复写入
