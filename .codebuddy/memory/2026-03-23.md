## 登录页问题复盘

用户反馈登录页两个问题：

**问题 1：`<t-input>` 组件用法不对**
- 原因：项目没有 TDesign uniapp 组件库 Skill，AI 用 Context7 查文档内容不准确，最终靠猜测写了 `:prefix-icon="{ name: 'user' }"` 这种错误用法
- 正确写法：`prefix-icon="user"`（直接传字符串），项目实际安装的是 `@tdesign/uniapp@0.7.3`
- 修复：需要改为正确的字符串 prop 写法

**问题 2：页面与设计稿不符**
- 顶部装饰图用了灰色占位图，设计稿是蓝色 3D 文件柜精美插画
- 输入框图标是 TDesign 默认灰色线条图标，设计稿是蓝色自定义图标
- 间距细节不够精确（表单区域与顶部图间距等）
- 根因在 UI 清单分析阶段（design-analysis），图标颜色/图片资源描述不够精确

**问题 3：API 响应处理方式错误**
- 原因：AI 写页面代码时没有去读 `src/http/core/request.ts` 和 `src/http/core/types.ts`，不了解实际请求封装逻辑
- 错误写法：用 `try/catch` 包裹，假设拦截器统一处理错误 toast
- 实际情况：`ApiRequest.request()` 内部已 try/catch，Promise **永远 resolve 不会 reject**，返回 `CoreResponse<T>` 包含 `{ code, data, msg, ok }` 四个字段，拦截器**不会自动 toast 错误**
- 正确写法：`if (res.ok) { 成功逻辑 } else { uni.showToast({ title: res.msg }) }`，不需要 try/catch
- 修复：已改为 `if/else` 分支

**结论**：需要创建 TDesign uniapp 组件库 Skill，固化常用组件的正确用法。create-page Skill 在写 API 调用代码时，应该强制要求先读取 `src/http/core/types.ts` 了解 CoreResponse 结构

## Skill 根因修复（2、3、4）

排查了登录页问题的根因链条，定位到 3 个 Skill 缺陷并完成修复：

### 修复 2：create-page Skill — 补充 UI 还原度检查
- **根因**：检查清单只检查代码规范，完全没有检查 UI 还原度（颜色、间距、图标对不对），AI "检查清单全过"就交差
- **改动**：场景一/场景二检查清单各增加 2 项——"UI 还原度验证"（逐区域对照 UI 清单确认样式）、"图标/颜色还原"（标注特定颜色的必须用 prop 或 CSS 覆盖实现）

### 修复 3：create-page Skill — 补充 API 调用强制规范
- **根因**：编写规范只说"API 从 @/http 导入"，没说要先读 `src/http/core/types.ts` 了解 CoreResponse 结构，AI 被 Spec 误导用了 try/catch
- **改动**：第 3 步编写规范新增"API 调用规范（强制）"段落——必须先读 types.ts、禁止 try/catch、必须用 if (res.ok) 判断，附正确/错误示例
- **同步改动**：场景一/场景二检查清单都增加了"API 调用方式"检查项

### 修复 4：create-proposal Skill — 补充 API 描述验证要求
- **根因**：Spec 中写了"拦截器已处理"这种错误描述，是因为 AI 在生成 Spec 时没有读代码验证，靠猜测编写
- **改动**：
  1. 核心原则第 12 条扩展——禁止自行编造 API 调用方式描述（如"拦截器已处理"、"try/catch 捕获异常"），API 调用方式只引用规范
  2. 新增第 13 条——涉及 API 行为的描述必须先读 `src/http/core/` 代码验证，禁止凭推测编写
  3. 模式 A 检查清单新增 2 项——验证 Spec 未编造 API 描述、API 行为描述已通过代码验证

## Skill 全链路根因修复（问题1-4 根源）

用户反馈登录页 4 个问题（占位图、v-model 无效、TDesign 组件用法错、样式不符），经全链路分析后定位到 3 个 Skill 的 5 处根因并全部修复：

### 1. design-analysis Skill — output-ui-inventory.md 模板
- **问题根源**："组件状态汇总"表格包含"实现方式建议"列（`:disabled`、`v-if`、`v-model` 等编码写法），导致 UI 清单越界写入代码实现

### 2. design-analysis Skill — SKILL.md 核心原则第5条
- **问题根源**：占位图术语模糊（"使用占位写法如 placeholder"），与 create-page Skill 的 `dummyimage.com` URL 格式不统一
- **修复**：明确图片占位用 `https://dummyimage.com/{宽}x{高}` 格式，图标占位用 `placeholder-icon`，禁止本地路径

### 3. design-analysis Skill — workflow-element-extraction.md
- **问题根源**：图片/图标表格模板没有"占位方式"列，示例也不含 dummyimage URL
- **修复**：新增"占位方式"列，示例改为完整 dummyimage URL，禁止 `placeholder-image` 抽象标记
- **同步修改**：output-ui-inventory.md 模板的图片/图标表格也加上"占位方式"列

### 4. create-proposal Skill — output-spec.md + SKILL.md
- **问题根源**：Spec 的"特殊样式要点"被当作 UI 清单的摘要版，大量样式细节丢失
- **修复**：
  - output-spec.md 模板的"特殊样式要点"加注释说明只列非标准样式提醒，禁止摘抄常规数值，并加 ⚠️ 提醒执行时必须逐区域对照 UI 清单
  - SKILL.md 第3步新增第8条规则，明确禁止摘抄 UI 清单常规间距/字号/颜色数值

### 5. create-page Skill — SKILL.md
- **问题根源**：TDesign 文档查阅是建议性的而非强制的；UI 还原度检查缺乏具体 diff 步骤
- **修复**：

  - 两个场景的 TDesign 检查项都更新为"已逐个查阅文档确认（特别是值绑定方式）"
