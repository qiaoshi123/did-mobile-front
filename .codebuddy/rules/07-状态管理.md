# 状态管理规范

## 基础规范

- 状态管理统一使用 **Pinia**，禁止其他方案
- 统一使用 **Composition API 风格**（setup 函数形式），禁止 Options API
- 每个 Store 一个文件，文件名 kebab-case，放在 `src/store/` 目录下

## Store 组织

- 按业务模块划分，每个模块单独一个文件
- 导出函数命名 `useXxxStore`（如 `useUserStore`、`useAuthStore`）
- 所有 Store 在 `src/store/index.ts` 中统一导出
- Store 中不要直接耦合具体 UI 组件，只存纯数据与业务行为

## 目录结构

- `src/store/index.ts` — 统一导出入口
- `src/store/<module>.ts` — 按业务模块一个文件（如 `user.ts`、`auth.ts`、`counter.ts`）

## 持久化

- 需要持久化的 Store 设置 `persist: true`（底层使用 uni storage）
- 使用 `pinia-plugin-persistedstate` 插件
- 仅对确实需要跨页面/刷新保留的数据开启持久化，不要滥用

## Store 解构规则

- 解构 Store 的 **state（属性）和 getters（计算属性）** 时，必须使用 `storeToRefs()`（从 `pinia` 导入），否则会丢失响应式
- 解构 Store 的 **actions（方法）** 时，直接从 store 实例解构即可，不需要 `storeToRefs`
- `storeToRefs()` 返回的是 `Ref` 对象：在 `<template>` 中自动解包，无需 `.value`；在 `<script>` 中必须通过 `.value` 访问
- 禁止对 store 实例直接使用 ES 解构获取 state/getters（如 `const { count } = store`），这会导致值失去响应式

## 禁止事项

| 序号 | 禁止行为 | 说明 |
|------|---------|------|
| 1 | 禁止使用 Options API 风格 | 统一使用 Composition API（setup 函数） |
| 2 | 禁止在 `src/store/` 以外存放 Store | 保持目录结构一致 |
| 3 | 禁止 Store 直接耦合 UI 组件 | Store 只存数据和业务逻辑 |
| 4 | 禁止直接解构 store 的 state/getters | 必须使用 `storeToRefs()` 保持响应式 |
