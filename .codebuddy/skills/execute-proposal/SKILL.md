# Spec 执行引擎

## 核心定位

**execute-proposal 是 Spec 提案的执行引擎**，负责读取已定稿的 Spec 文档，按照任务清单逐个调用下游 Skill 完成开发工作。

**它只负责"执行"，不负责"规划"。** 规划（分析需求、拆解任务、输出 Spec）由 `create-proposal` Skill 完成。

### 职责边界

| 职责 | 说明 |
|------|------|
| ✅ 读取 Spec 文档 | 从 `docs/specs/` 目录下找到目标 Spec |
| ✅ 按任务类型执行 | Skill 任务通过 `use_skill` 调用；直接执行任务直接编写代码 |
| ✅ 传递任务需求 | 将 Spec 中每个任务的需求描述作为 Skill 的输入上下文 |
| ✅ 同步任务状态 | 每完成一个任务，更新 Spec 文件中该任务状态为 ✅ |
| ❌ 不分析需求 | 需求分析是 `create-proposal` 的职责 |
| ❌ 不拆解任务 | 任务拆解是 `create-proposal` 的职责 |
| ❌ 不修改任务内容 | Spec 中的任务需求是确定的，不可擅自修改 |

---

## 工作流程

```
用户触发执行（如"执行 Spec"、"开始开发"）
    ↓
第1步：定位 Spec 文档
    ↓
第2步：校验 Spec 状态
    ↓
第3步：逐个执行任务
    ↓
第4步：完成收尾
```

---

## 第1步：定位 Spec 文档

### 查找策略

1. **用户指定了 Spec 名称** → 直接读取 `docs/specs/<name>-spec.md`
2. **用户未指定** → 扫描 `docs/specs/` 目录，找到状态为 `🟢 定稿` 的 Spec 文件
3. **多个定稿 Spec** → 列出所有定稿 Spec，让用户选择执行哪个
4. **没有定稿 Spec** → 告知用户需要先通过 `create-proposal` 生成并定稿 Spec

---

## 第2步：校验 Spec 状态

### 前置检查

- **状态必须为 `🟢 定稿`**：只有定稿状态的 Spec 才能执行
- **`🔶 草稿`**：告知用户 Spec 还在决策确认中，需要先完成定稿
- **`🔵 执行中`**：需进一步判断是"断点续执行"还是"变更后重执行"：
  - 检查「变更记录」中是否有 v2 或更高版本 + 是否同时存在 ✅ 和 ⬜ 任务
  - **是** → 进入"变更后重执行"模式（见下方独立章节）
  - **否** → 进入"断点续执行"模式
- **`✅ 已完成`**：告知用户该 Spec 已全部执行完毕

### 状态更新

- **`🟢 定稿` → `🔵 执行中`**：首次执行时更新
- **`🔵 执行中`**：断点续执行和变更后重执行无需更新状态（已经是执行中）

---

## 第3步：逐个执行任务

### 任务类型判断

Spec 中的每个任务会标注执行方式。根据标注决定执行路径：

| 任务标注 | 执行方式 | 示例 |
|---------|---------|------|
| **Skill 任务**（标注了 Skill 名称） | 通过 `use_skill("<skill-name>")` 加载 Skill 执行 | 创建新页面、新建 Store、新建 Hook、新建 Util 等 |
| **直接执行任务**（标注为"直接执行"） | 直接编写/修改代码，无需调用 Skill | 修改常量值、往配置文件加一行、简单的代码改动等 |

### 执行协议

#### 路径 A：Skill 任务

对标注了 Skill 名称的任务，**必须按以下步骤执行**：

1. **调用 Skill**：使用 `use_skill("<skill-name>")` 加载该任务指定的 Skill
2. **传入需求**：将该任务的"需求描述"作为 Skill 的输入上下文
3. **Skill 自治**：Skill 自行执行其内部流程，execute-proposal 不干预 Skill 的内部决策
4. **标记完成**：任务完成后，更新 Spec 文件中该任务状态为 ✅
5. **进入下一个**：继续下一个任务

**⚠️ Skill 任务绝对禁止跳过 Skill 直接编写代码。** 每个下游 Skill 内部都有详细的执行规则和约束（文件位置、代码结构、导出注册等），跳过 Skill 会导致这些规则失效。

#### 路径 B：直接执行任务

对标注为"直接执行"的任务：

1. **理解需求**：读取 Spec 中该任务的需求描述
2. **直接执行**：按照需求描述直接编写/修改代码
3. **标记完成**：更新 Spec 文件中该任务状态为 ✅
4. **进入下一个**：继续下一个任务

### 通用约束

| 规则 | 说明 |
|------|------|
| **严格顺序** | 按 Spec 中的执行顺序执行，不跳跃、不并行 |
| **一次一个** | 一个任务完成后才执行下一个 |
| **状态同步** | 每完成一个任务，立即更新 Spec 文件中的任务状态 |
| **不修改需求** | 严格按照 Spec 中的任务需求描述执行，不擅自增减需求内容 |

### 传入需求的格式（Skill 任务）

调用 Skill 后，向其传入以下信息：

```
任务：<任务名称>
Spec 来源：<Spec 文件路径>
需求描述：<Spec 中该任务的完整需求描述>
已有资源：<Spec 中该任务列出的可复用资源>
预期产出：<Spec 中该任务的预期产出文件>
```

---

## 第4步：完成收尾

所有任务执行完毕后：

1. **更新 Spec 状态**：将 Spec 文档状态从 `🔵 执行中` 更新为 `✅ 已完成`
2. **输出执行摘要**：
   - 列出所有已完成的任务
   - 列出所有产出的文件
   - 如有执行过程中遇到的问题或需要注意的事项，一并列出
3. **自动归档**：执行归档流程（见下方"自动归档"章节）
4. **结束**：execute-proposal 工作完成

---

## 自动归档

**触发时机**：Spec 所有任务执行完毕、状态更新为 `✅ 已完成` 后，**自动执行归档**，无需用户手动触发。

### 归档流程

1. **移动 Spec 文件**：将 `docs/specs/<name>-spec.md` 移动到 `docs/specs/.archive/<name>-spec.md`
2. **更新时间线**：在 `docs/specs/.archive/TIMELINE.md` 的表格中**追加一行记录**

### 时间线记录格式

在 `TIMELINE.md` 的表格末尾追加：

```
| <当前日期 YYYY-MM-DD> | <Spec 标题> | <任务数量> | <涉及的文件列表，逗号分隔> | <Spec 中的决策摘要，简要概括> |
```

**字段说明**：
- **完成日期**：执行完成当天的日期
- **Spec 名称**：Spec 文档的一级标题
- **任务数**：该 Spec 中的任务总数
- **涉及文件**：所有任务产出/修改的文件路径，用 `、` 分隔
- **决策摘要**：从 Spec 的"决策记录"章节中提炼，一句话概括

### 归档约束

| 规则 | 说明 |
|------|------|
| **只归档已完成的 Spec** | 状态为 `✅ 已完成` 才能归档 |
| **归档文件保持原样** | 移动到 `.archive/` 后不修改 Spec 文件内容 |
| **时间线只追加** | 只在表格末尾新增行，不修改已有记录 |
| **归档文件永久保留** | `.archive/` 中的文件不会被删除 |

---

## 断点续执行

如果 Spec 状态为 `🔵 执行中`（之前执行被中断），execute-proposal 会：

1. 读取 Spec 中各任务的状态标记
2. 跳过已标记为 ✅ 的任务
3. 从第一个未完成的任务开始继续执行

---

## 变更后重执行

### 触发条件

当 Spec 满足以下**全部条件**时，判定为"变更后重执行"：

1. Spec 状态为 `🔵 执行中`
2. Spec 的「变更记录」章节中存在 v2 或更高版本的变更记录（说明 Spec 经历过变更）
3. 既有 ✅ 已完成的任务，又有 ⬜ 待执行的任务（说明部分任务需要重新执行）

### 与断点续执行的区别

| 维度 | 断点续执行 | 变更后重执行 |
|------|----------|------------|
| **原因** | 执行过程被意外中断 | Spec 变更后部分任务需要重做 |
| **⬜ 任务的文件** | 不存在（还没创建） | **可能已存在**（之前已执行过，现在需要修改） |
| **传入 Skill 的上下文** | 正常传入需求 | **额外告知 Skill：这是变更后重执行，相关文件已存在需要修改而非新建** |

### 执行流程

1. **检测变更记录**：读取 Spec 的「变更记录」章节，找到最新一次变更记录，提取"受影响任务"列表
2. **确定重执行任务**：状态为 ⬜ 的任务即为需要重新执行的任务（通常与变更记录中的"受影响任务"一致）
3. **跳过已完成任务**：与断点续执行一致，跳过 ✅ 任务
4. **执行待执行任务**：对每个 ⬜ 任务，按以下方式执行：

#### 传入需求的格式（变更后重执行）

调用 Skill 后，向其传入以下信息（比正常执行多了"变更说明"）：

```
任务：<任务名称>
Spec 来源：<Spec 文件路径>
执行模式：变更后重执行（该任务之前已执行过，相关文件已存在，请以"修改"而非"新建"模式执行）
变更说明：<从最新变更记录中提取该任务相关的变更点>
需求描述：<Spec 中该任务的完整需求描述（已是变更后的最新版本）>
已有资源：<Spec 中该任务列出的可复用资源>
预期产出：<Spec 中该任务的预期产出文件>
```

**关键区别**：增加了 `执行模式` 和 `变更说明` 两个字段，让下游 Skill 明确知道：
- 目标文件已存在，应该读取现有文件后进行修改，而不是从零创建
- 具体需要修改什么（变更说明）

### 约束

| 规则 | 说明 |
|------|------|
| **不重新执行已完成任务** | 只执行状态为 ⬜ 的任务，不碰 ✅ 的任务 |
| **变更说明必须准确** | 从 Spec 变更记录中提取，不自行推断 |
| **其余规则与正常执行一致** | 执行顺序、状态同步、Skill 调用方式等均不变 |

## 可调度的下游 Skill

以下是 execute-proposal 可以调用的下游 Skill（与 `create-proposal` 中定义的一致）：

| Skill | 职责 |
|-------|------|
| create-route | 页面路由配置（pages.json + 路由常量） |
| create-api | 接入 API 接口 |
| create-store | 创建/修改 Pinia Store |
| create-hook | 创建/修改 Composable |
| create-util | 创建/修改工具函数 |
| create-sfc | 创建/修改完整的 Vue SFC 文件（template + script + style） |

## 检查清单

### 执行前

- [ ] 已找到目标 Spec 文档
- [ ] Spec 状态为 `🟢 定稿` 或 `🔵 执行中`（断点续执行 / 变更后重执行）
- [ ] 已正确判断执行模式：首次执行 / 断点续执行 / 变更后重执行
- [ ] （变更后重执行）已从变更记录中提取受影响任务和变更说明
- [ ] 已将 Spec 状态更新为 `🔵 执行中`（如尚未更新）

### 执行中（每个任务）

- [ ] 已正确判断任务类型（Skill 任务 / 直接执行任务）
- [ ] Skill 任务：**已通过 `use_skill` 加载该任务指定的 Skill**（未跳过）
- [ ] Skill 任务：已将完整的任务需求传递给 Skill
- [ ] （变更后重执行）Skill 任务：已在传入需求中包含 `执行模式` 和 `变更说明` 字段
- [ ] 直接执行任务：已按 Spec 需求描述完成代码修改
- [ ] 任务产出符合预期
- [ ] 已更新 Spec 文件中该任务状态为 ✅

### 执行后

- [ ] 所有任务状态已更新为 ✅
- [ ] Spec 文档状态已更新为 `✅ 已完成`
- [ ] 已输出执行摘要
