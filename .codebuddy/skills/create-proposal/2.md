# 智能提案编排系统

## 核心定位

**create-proposal 是整个开发流程的"大脑"**，根据用户的任意诉求，动态推理出需要哪些任务、什么顺序、调用哪些 Skill。

**不预设场景，不穷举分类，根据诉求本身去思考。**

### 三条核心原则

1. **用户确认**：所有资源创建决策都需要用户确认，AI 不自主决策
2. **代码真相源**：每次都扫描代码库获取最新状态，不靠记忆
3. **复用优先**：能复用现有资源就复用，避免重复建设

### 决策规则

```
用户明确说明需要 → 直接生成任务
用户未说明 → AI 分析是否需要 → 输出决策清单 → 用户确认 → 生成任务
```

---

## 可调度的下游 Skill

| Skill | 职责 | 典型产出 |
|-------|------|---------|
| create-api | 接入 API 接口 | `src/http/services/` 下的服务文件 |
| create-store | 创建/修改 Pinia Store | `src/store/` 下的 Store 文件 |
| create-hook | 创建/修改 Composable | `src/hooks/` 下的 Hook 文件 |
| create-util | 创建/修改工具函数 | `src/utils/` 下的工具文件 |
| create-ui | 创建 UI（模板+样式） | 页面/组件的 `<template>` + `<style>` |
| create-logic | 创建逻辑代码 | 页面/组件的 `<script setup>` |
| design-analysis | 分析设计稿 | UI 清单文档 |

**AI 根据诉求自由组合这些 Skill，不限定固定的组合方式。**

---

## 工作流程

```
用户提供诉求
    ↓
第1步：理解诉求
    ↓
第2步：按需扫描代码库
    ↓
第3步：分析（复用 + 抽离）
    ↓
第4步：输出决策清单（⚠️ 必须暂停，等待用户确认）
    ↓
第5步：生成提案文档
    ↓
第6步：交付确认（⚠️ 必须暂停，等待用户确认）
    ↓
第7步：执行任务
```

---

## 第1步：理解诉求

**目的**：搞清楚用户到底要做什么，而不是急着分类。

### 需要想清楚的问题

1. **用户要达成什么目标？**（不是"输入了什么文档"，而是"要什么结果"）
2. **涉及哪些资源类型？**（页面？组件？Hook？Store？API？Util？）
3. **有没有设计稿？** → 有则先调用 `design-analysis` 生成 UI 清单
4. **涉及几个页面/模块？** → 多页面联动需要先分析共享资源
5. **是新建还是修改？** → 修改需要先读取现有代码

### 关键原则：输入源决定分析边界

- **有设计稿** → 涉及 UI 层面（组件复用、组件抽离）
- **无设计稿** → 不涉及 UI 层面，不分析组件复用/抽离
- **有交互文档/API文档** → 涉及逻辑层面（Hook/Store/Util 抽离）
- **无交互文档** → 不涉及逻辑层面

**没有提供的信息就不要去分析，边界要清晰。**

---

## 第2步：按需扫描代码库

**目的**：只扫描与诉求相关的资源，建立复用基础。

### 扫描原则

**不是每次都全量扫描。** 根据第1步分析出的涉及资源类型，只扫描需要的部分：

| 涉及资源 | 需要扫描 |
|---------|---------|
| API | `src/http/services/` 下相关服务 |
| Store | `src/store/index.ts` |
| Hook | `src/hooks/index.ts` |
| Util | `src/utils/index.ts` |
| 组件 | `src/components/index.ts` |
| 页面 | `src/pages/` 目录结构 + `src/pages.json` |
| 路由 | `src/constants/routes.ts` + `src/pages.json` |

### 始终扫描

无论什么诉求，以下始终需要扫描（因为可能存在复用机会）：

- `src/pages.json`（路由全局配置）
- `src/constants/routes.ts`（路径常量）

---

## 第3步：分析（复用 + 抽离）

**目的**：基于扫描结果，分析哪些资源可以复用、哪些逻辑值得抽离。

### 复用分析

对于诉求涉及的每种资源，检查现有代码库是否已有可复用的：

- **完全匹配** → 直接复用
- **部分匹配** → 评估是否可扩展现有资源
- **无匹配** → 需要新建

组件复用时，遵循 `04-组件规范` 的优先级：本地组件 > TDesign 组件 > 新建。

### 抽离分析

**核心原则：只分析与诉求相关的抽离维度。**

- **涉及 UI（有设计稿）** → 分析组件抽离
- **涉及逻辑（有交互/API文档）** → 分析 Hook / Util / Store 抽离
- **两者都有** → 全部分析

#### 抽离判断通用原则

| 维度 | 建议抽离 | 不建议抽离 |
|------|---------|----------|
| 复用性 | 2+ 个页面/模块使用 | 仅单处使用 |
| 复杂度 | 逻辑复杂、有状态管理 | 逻辑简单、几行代码 |
| 独立性 | 可独立运作、职责单一 | 与业务强耦合 |

#### 资源类型判断

- **有响应式状态（ref/reactive）** → Hook
- **纯函数、无副作用** → Util
- **全局状态、跨页面共享** → Store
- **UI 结构复杂、可复用** → 组件

---

## 第4步：输出决策清单（⚠️ 必须暂停）

**这是最关键的步骤！AI 必须在此步骤暂停，等待用户确认所有决策。**

### 决策清单内容

根据诉求实际情况，输出以下内容（**只输出与诉求相关的部分，不相关的不要出现**）：

#### 一、任务拆分方案

列出 AI 分析后认为需要执行的任务，每个任务对应一次 Skill 调用。

#### 二、资源复用决策

列出诉求涉及的资源需求，以及现有代码库中是否有可复用的。

#### 三、抽离决策

对于用户未明确说明、但 AI 分析认为有抽离价值的资源，列出决策问题让用户确认。

#### 四、已确定的信息

列出可以确定的信息（已有的 API、已有的 Hook/Util/Store 等）。

### 绝对禁止

- ❌ 跳过此步直接输出定稿提案
- ❌ AI 自行决定任务拆分方案不问用户
- ❌ 把 AI 的建议直接当确定方案写入提案
- ❌ 输出与诉求无关的分析（如没有设计稿却分析组件复用）

### 用户回复选项

```
1. 💬 逐条回复 → 根据回答生成定稿提案
2. ⏭️ 全部按建议来 → 按建议列填入提案
3. 🔀 部分回复，部分待补充
4. ✏️ 整体调整
```

**等待用户回复后，进入下一步。**

---

## 第5步：生成提案文档

### 文档保存位置

```
docs/specs/<name>-spec.md
```

### 文档结构

提案文档包含以下部分（**根据诉求裁剪，不相关的章节不写**）：

1. **需求摘要** — 用户要做什么、涉及哪些页面/模块
2. **任务清单** — 每个任务 = 一次 Skill 调用，按依赖排序
3. **任务详情** — 每个任务的需求描述、使用的已有资源、预期产出
4. **资源方案总览** — 涉及的组件/Store/Hook/Util/API 汇总
5. **决策记录** — 用户确认的所有决策
6. **执行顺序** — 任务执行的先后顺序
7. **变更记录** — 文档变更历史

### 任务排序原则

根据任务之间的实际依赖关系排序：
- **被依赖的任务排前面**（如 Store 被页面逻辑依赖，则 Store 先建）
- **无依赖关系的任务可以并行**
- **不预设固定的 Skill 顺序**，根据实际情况决定

### 文档状态

| 标记 | 含义 |
|------|------|
| `🔶 草稿` | 正在确认决策中 |
| `🟢 定稿` | 所有决策已确认，可以执行 |
| `🔵 执行中` | 任务正在逐个执行 |
| `✅ 已完成` | 所有任务执行完毕 |

---

## 第6步：交付确认（⚠️ 必须暂停）

向用户展示定稿摘要，包含：
- Spec 文件路径
- 任务数量
- 待补充信息（如有）

**等待用户回复。绝对禁止生成 Spec 后直接开始开发。**

---

## 第7步：执行任务

### 执行协议

对每个任务：

1. **调用 Skill**：使用 `use_skill("<skill-name>")` 加载对应 Skill
2. **传入需求**：将该任务的"需求描述"作为 Skill 的输入上下文
3. **Skill 自治**：Skill 自行执行其内部流程
4. **标记完成**：任务完成后，更新 Spec 文件中该任务状态为 ✅
5. **进入下一个**：继续下一个任务

### 执行规则

- **严格顺序**：按 Spec 中执行顺序执行，不跳跃
- **一次一个**：一个任务完成后才执行下一个
- **Skill 自主**：不干预 Skill 的内部决策过程
- **状态同步**：每完成一个任务，同步更新 Spec 文件中的任务状态

---

## 提案文档管理

### 代码作为真相源

每次生成提案都全新分析代码库，不依赖历史提案文档。

### 归档

- **活跃文档**：`docs/specs/<name>-spec.md`
- **归档文档**：`docs/specs/.archive/<name>/<version>-tasks.md`
- **触发**：手动归档，或活跃文档中已完成任务超过 5 个时自动归档

---

## 职责边界

**create-proposal 只说"做什么"，下游 Skill 决定"怎么做"。**

- ✅ 分析诉求，识别需要哪些资源
- ✅ 输出决策清单，等待用户确认
- ✅ 编排任务，确定执行顺序
- ❌ 不教下游 Skill 怎么实现
- ❌ 不替下游 Skill 做抽离以外的技术决策
