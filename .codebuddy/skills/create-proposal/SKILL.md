# 生成 Spec 实施计划

## 定位

Spec 是**开发流程控制文档**，核心价值：

1. **需求不遗漏** — 记录页面的完整需求清单
2. **开发不脱节** — 任务拆分 + 依赖顺序 + 状态追踪
3. **决策可追溯** — 记录用户确认过的每个决策
4. **可复盘归档** — 页面的决策档案，看 Spec 就知道做了什么、为什么这么做

### Spec 不是什么

- ❌ **不是实现指南** — 不教 AI 怎么写代码、怎么设计 Store/Hook 的字段
- ❌ **不是 AI 决策记录** — AI 不做业务决策，所有决策由用户确认
- ❌ **不是代码的映射** — 开发完成后代码是真相源，Spec 是决策档案

### 核心分工

| 角色 | 职责 | 不做 |
|------|------|------|
| **Spec（本技能）** | 需求清单、任务拆分、依赖顺序、决策记录、状态追踪 | 不写实现细节 |
| **create-xxx Skills** | 根据任务需求描述 + 项目规范，自行决定具体实现 | — |
| **Rules** | 约束每个 Skill 的输出必须符合规范 | — |

> **一句话：Spec 说"做什么"，Skill 决定"怎么做"，Rule 约束"做的对不对"。**

---

## Spec 管理的边界

**Spec 管理的是页面级需求**，以下情况**不走 Spec 流程**：

| 场景 | 处理方式 |
|------|---------|
| 全局改动（如给所有页面加 loading 遮罩） | 直接执行，不写 Spec |
| 基础设施改动（如 HTTP 拦截器加 token 刷新） | 直接执行，不写 Spec |
| 公共组件改动（如 DidCard 新增 prop） | 直接执行，不写 Spec |
| 页面级需求（新建/修改某个页面） | **必须走 Spec** |

> 判断标准：**这个改动是以某个页面为中心的需求吗？** 是 → 走 Spec；否 → 直接改。

---

## ⚠️ 两种工作模式

### 模式 A：全新生成 Spec

**触发条件**：用户提供的是一份新的 UI 清单，且 `docs/specs/` 下没有对应的 Spec 文件。

**流程**：走完整的第 0~5 步。

### 模式 B：增量更新已有 Spec

**触发条件**：用户要求补充/更新已有 Spec 的信息（如补充 API 文档、补充交互说明、页面迭代新增功能等），且 `docs/specs/` 下已有对应的 Spec 文件。

**流程**：
1. 读取已有 Spec 文件
2. 读取用户新提供的物料
3. **只更新受影响的章节和任务**，不推翻已有内容
4. 已标记为 ✅ 的任务不修改
5. 原"⚠️ 待补充"的内容用新信息填充
6. 如有新增任务，插入到正确的依赖位置
7. **在"变更记录"章节追加本次变更条目**
8. 输出更新后的完整 Spec 文件
9. 向用户展示变更摘要，等待确认

## 核心原则

### AI 的角色边界

1. **AI 是流程编排者，不是设计决策者**。AI 负责整理需求、拆分任务、发现盲点，但**所有业务决策必须由用户确认**
2. **AI 不猜、不推断、不发挥**。用户没说的，问；用户说了的，如实记录
3. **AI 不教下游 Skill 怎么做**。Spec 中的任务描述只说"需要什么"，具体实现由 Skill 根据项目规范自行完成

### 任务拆分原则

4. **一个任务 = 一次 Skill 调用**
5. **已有资源不生成任务**。通过扫描项目确认已有的组件/API/Store/Hook/Util，在使用方任务中标注"直接使用"
6. **依赖关系必须正确**。被依赖的任务排在前面
7. **先基础后组装**。顺序：API → Store → Hook/Util → 组件 → 页面

### 内容原则

8. **API 使用清单是强制项**。用户必须提供，AI 不猜
9. **Spec 禁止包含代码块**。所有信息用表格 + 文字
10. **Spec 禁止包含实现细节**。不写字段定义、不写函数签名、不写样式数值、不写校验正则
11. **UI 信息引用原则**。create-page 任务引用 UI 清单文件路径，不复制样式细节
12. **遵循 HTTP 响应约定**。API 调用描述标注"遵循 `05-API规范` 的响应约定"即可

## 工作流程（模式 A：全新生成）

> **流程总览**：第 0 步资源扫描 → 第 1 步 API 确认 → **第 2 步需求分析 + 任务拆分草稿（必须暂停，向用户确认全部决策）** → 第 3 步输出定稿 → 第 4 步交付确认 → 第 5 步完成

### 第 0 步：扫描项目现有资源

> ⚠️ 每次生成 Spec 前都必须执行，不可跳过。

**操作：**

1. 读取 UI 清单文档（重点关注第七章"给 create-proposal 的摘要"）
2. 读取 `src/components/index.ts`（已有通用组件）
3. 读取 `src/store/index.ts`（已有 Store）
4. 读取 `src/hooks/index.ts`（已有 Hooks）
5. 读取 `src/utils/index.ts`（已有工具函数）
6. 读取相关服务的 `src/http/services/<service>/index.ts`（已有 API）
7. 读取 `src/pages.json` 和 `src/constants/routes.ts`（已有路由）
8. 检查 `src/pages/` 下是否已有同名页面

---

### 第 1 步：API 使用清单确认

> 此步骤只处理 API 相关确认。

**前置条件**：用户已提供 API 使用清单。如未提供，先索要：

```
⚠️ 生成 Spec 前，我需要你提供该页面要调用的接口列表。
格式如：登录 → didappLogin
如果你不确定有哪些接口可用，我可以先列出项目已有的全部 API 函数供你选择。
```

**确认流程**：逐个去 `src/http/services/` 查找用户指定的接口，按查找结果走三分支：

| 项目中状态 | 处理 |
|-----------|------|
| ✅ 已找到 | 问用户：直接使用 or 需要更新？ |
| ❌ 未找到 | 问用户：请补充接口文档 or 不需要了？ |

合并为一张表一次性确认：

```
🔍 API 使用清单核查结果：

| # | 接口 | 项目中状态 | 需要你确认 |
|---|------|----------|----------|
| 1 | didappLogin | ✅ 已找到 | 直接使用 or 需要更新？ |
| 2 | sendSmsCode | ❌ 未找到 | 请补充接口文档 or 不需要了？ |
```

**等待用户回复后，进入第 2 步。**

---

### 第 2 步：需求分析 + 草稿确认（⚠️ 核心关卡，必须暂停）

> **这是整个流程中最关键的步骤。AI 在这一步做需求分析和任务拆分，但所有决策必须以草稿形式提交给用户确认，不允许直接定稿。**

**操作：**

#### 2.1 需求分析（AI 内部进行，不直接输出）

基于 UI 清单 + API 确认结果 + 用户提供的交互说明，分析以下维度：

- 页面基本信息（路由、导航栏、是否 tabBar 等）
- 需要新建的资源（API / Store / Hook / Util / 组件）
- 可直接使用的已有资源
- 用户已明确的交互逻辑
- **用户未明确但开发时必须决策的点**（决策盲点）

#### 2.2 输出草稿 + 决策确认清单

**将分析结果以"草稿 + 待确认事项"的形式输出给用户。格式：**

```
📋 Spec 草稿分析完成，以下内容需要你确认：

### 一、任务拆分方案

我分析认为这个页面需要以下任务（请确认是否同意）：

| # | 任务 | 触发 Skill | 理由 | 你的决定 |
|---|------|-----------|------|---------|
| T1 | 新建 useUserStore | create-store | 登录成功后需要存储 token 和用户信息 | ✅同意 / ❌不需要 / ✏️调整？ |
| T2 | 新建 useLogin Hook | create-hook | 登录表单逻辑可复用 | ✅同意 / ❌直接写页面里 / ✏️调整？ |
| T3 | 创建登录页 | create-page | — | ✅同意 |

### 二、交互决策（用户未明确，需要确认）

| # | 决策问题 | 为什么需要决策 | 我的建议（仅供参考） |
|---|---------|--------------|-------------------|
| D1 | 登录成功后跳转哪个页面？ | 需要确定目标路由 | 跳转首页 |
| D2 | 登录成功后 token 存哪里？需要全局缓存用户信息吗？ | 影响是否新建 Store | 存 localStorage + Store |
| D3 | 密码输入错误提示方式？ | toast / 行内提示 / 弹窗 | toast |
| D4 | "创建企业账号"和"忘记密码"点击后跳哪？ | 目标页是否已开发 | 预留空跳转 |

### 三、已确定的信息

- API：didappLogin（✅ 直接使用）
- TDesign 组件：t-input、t-button（✅ 直接使用）
- 已有 Hook：useLoading、useFormValidation（✅ 直接使用）
- 路由：/pages/login/index（主包，custom 导航栏）

### 你可以：

1. 💬 **逐条回复** → 我根据你的回答生成定稿 Spec
2. ⏭️ **全部按建议来** → 我按"建议"列填入 Spec
3. 🔀 **部分回复，部分待补充** → 你确认的写入，其余标"⚠️ 待补充"
4. ✏️ **整体调整** → 告诉我你的想法
```

**绝对禁止的行为**：
- ❌ 跳过此步直接输出定稿 Spec
- ❌ AI 自行决定任务拆分方案不问用户
- ❌ AI 自行决定交互逻辑不问用户（如自行决定"登录后跳首页"）
- ❌ 把 AI 的建议直接当确定方案写入 Spec

**等待用户回复后，进入第 3 步。**

---

### 第 3 步：输出定稿 Spec 文档

**触发条件**：用户在第 2 步回复了确认。

**操作：**

1. 根据用户确认的决策，生成 Spec 文档到 `docs/specs/<名称>-spec.md`
2. 文档结构按 `rules/output-spec.md` 模板
3. 文件头状态标注为 `🟢 定稿`
4. 用户确认的信息正常写入
5. 用户说"后面再补"的标注"⚠️ 待补充"
6. **任务详情只写需求描述（做什么），不写实现指导（怎么做）**
7. **create-page 任务的 UI 信息引用 UI 清单路径，不复制样式数值**

---

### 第 4 步：交付确认（⚠️ 必须暂停）

**操作：向用户展示定稿摘要**

```
📋 Spec 定稿已生成：`docs/specs/<名称>-spec.md`

### 摘要
- **页面**：<名称>（<路径>）
- **任务数**：X 个
- **待补充信息**：<列出所有 ⚠️ 项，没有则写"无">

### 你可以：
1. ✅ **确认无误，开始执行** → 按任务顺序逐个执行
2. ✏️ **需要修改** → 告诉我哪里需要调整
3. ➕ **补充信息后再执行**
4. 📖 **先看看 Spec 内容**
```

**等待用户回复。绝对禁止生成 Spec 后直接开始开发。**

---

### 第 5 步：开始执行

**触发条件**：用户在第 4 步确认"开始执行"。

**操作**：按 Spec 中的执行顺序逐个执行任务。

#### 执行协议（Spec → Skill 衔接）

对每个任务，按以下步骤执行：

1. **调用 Skill**：使用 `use_skill("<触发 Skill 名>")` 加载对应 Skill（如 `use_skill("create-store")`）
2. **传入需求**：Skill 加载后，将该任务的"需求描述"作为 Skill 的输入上下文。告知 Skill："根据以下需求执行：" + Spec 中该任务的需求描述
3. **Skill 自治**：Skill 自行执行其内部流程（包括第 0 步前置检查），如果 Skill 需要向用户确认信息，正常暂停等待
4. **标记完成**：任务完成后，回到 Spec 更新该任务状态为 ✅
5. **进入下一个**：继续下一个任务，重复步骤 1-4

#### 执行规则

- **严格顺序**：按 Spec 中"七、执行顺序"章节的顺序执行，不跳跃
- **一次一个**：一个任务完成后才执行下一个
- **Skill 自主**：不干预 Skill 的内部决策过程，Skill 根据需求描述 + 项目规范自行决定实现方式
- **异常处理**：如果某个 Skill 执行失败或被用户中止，暂停执行，向用户报告，等待用户指示
- **状态同步**：每完成一个任务，同步更新 Spec 文件中的任务状态（⬜ → ✅）和文件头状态（首个任务开始时改为 🔵 执行中，全部完成后改为 ✅ 已完成）

## 任务拆分规则

### 生成任务的情况

| 需求 | 任务 | 触发 Skill |
|------|------|-----------|
| 需要新增 API 接口 | "接入 xxx 接口" | create-api |
| 需要新建 Store | "创建 xxxStore" | create-store |
| 需要新建 Hook | "创建 useXxx Hook" | create-hook |
| 需要新建工具函数 | "创建 xxx 工具函数" | create-util |
| 需要新建组件 | "创建 xxx 组件" | create-component |
| 需要创建/修改页面 | "创建 xxx 页面" | create-page |

### 不生成任务的情况

- 已有的 API / Store / Hook / Util / 组件 → 在使用方任务中标注"直接使用"
- TDesign 组件 → 在页面任务中说明
- 路由配置 → 由 create-page 内自动完成

### 任务排序

```
1. create-api（最底层）
2. create-store
3. create-hook / create-util
4. create-component
5. create-page（最顶层，组装所有）
```

## 任务详情的写法原则

> ⚠️ **Spec 中的任务详情只描述"需要什么"，不描述"怎么实现"。**

### ✅ 正确示例

```
### T1：新建用户 Store（useUserStore）

**触发 Skill**：create-store
**需求描述**：
- 存储登录态相关数据（token、用户信息）
- 提供登录/登出的状态切换能力
- 需要持久化

**依赖**：无
**预期产出**：src/store/user.ts
```

### ❌ 错误示例（Spec 越权，教 Skill 怎么做）

```
### T1：新建用户 Store（useUserStore）

**执行指令**：
管理的状态：
- token: string，默认值 ''
- userInfo: DidappUserInfo | null，默认值 null
Actions：
- setAuth(token: string, userInfo: DidappUserInfo)：保存 token 和 userInfo
- clearAuth()：将 token 重置为 ''，userInfo 重置为 null
Getters：
- isLoggedIn: boolean — 返回 token !== ''
开启持久化，持久化字段：token、userInfo。
```

**区别**：正确示例说"存储登录态数据，需要持久化"；错误示例教 AI 定义了字段名、类型、默认值、方法签名 —— 这些是 create-store Skill 根据需求和项目规范自行决定的。

## API 使用清单规则（强制要求）

### 核心原则

**用户必须明确提供接口列表。AI 不猜、不推测。**

### 用户输入格式（任意一种）

- 功能 → 函数名：`登录 → didappLogin`
- 功能 → 接口地址：`登录 → POST /api/didapp/login`
- 直接列函数名：`didappLogin、didappAuthCodeVerify`

### 如果用户没提供

必须主动索要，不允许跳过或标注"待补充"。

### 三分支处理流程

用户提供清单后，逐个在 `src/http/services/` 查找：

| 状态 | 操作 |
|------|------|
| ✅ 找到了 | 问用户：直接使用 or 需要更新？ |
| ❌ 没找到 | 问用户：补充文档 or 不需要了？ |

确认结果：
- "直接使用" → 写入 Spec 的已有 API 清单，不生成任务
- "需要更新" → 生成 create-api 更新任务
- "补充了文档" → 生成 create-api 新建任务
- "不需要了" → 从清单移除

## Spec 状态标注规范

| 状态 | 标记 | 含义 |
|------|------|------|
| 草稿 | `🔶 草稿` | 正在确认决策中 |
| 定稿 | `🟢 定稿` | 所有决策已确认，可以执行 |
| 执行中 | `🔵 执行中` | 任务正在逐个执行 |
| 已完成 | `✅ 已完成` | 所有任务执行完毕 |

状态流转：草稿 → 定稿 → 执行中 → 已完成

## 变更记录规范

每份 Spec 文件末尾包含"变更记录"章节：

```
## 变更记录

### v1 — 初始创建（YYYY-MM-DD）
- 创建登录页 Spec

### v2 — 新增第三方登录（YYYY-MM-DD）
- 新增微信登录按钮
#### 关联变更
- `docs/specs/home-spec.md`：首页需同步适配
```

当变更记录超过 5 个版本时，归档旧版本到 `docs/specs/<名称>-spec-history.md`。

## 页面迭代流程

```
任何页面级改动需求
    ├── 有设计稿 → design-analysis → create-proposal 模式 B → 确认 → 执行
    └── 无设计稿（口述） → create-proposal 模式 B → 确认 → 执行
```

迭代规则：
1. 每次迭代只更新受影响的部分
2. 已完成任务（✅）保持不变
3. 每次追加变更记录

## 与其它技能的关系

```
design-analysis（眼睛）
       ↓ 产出 UI 清单
create-proposal（大脑）← API 文档 / 交互说明
       ↓ 产出 Spec（需求清单 + 任务拆分 + 决策记录）
┌──────┼──────┐
↓      ↓      ↓
create-api  create-store  create-component  ...（手）
       ↓
   create-page（最后组装）
```

- Spec 只说"做什么" → 每个 Skill 自行读规范决定"怎么做"
- Spec 不教 Skill 怎么实现 → Skill 有自己的模板和规范
