# 智能提案编排系统

## 核心定位

**create-proposal 是整个开发流程的"大脑"**，根据用户的任意诉求，动态推理出需要哪些任务、什么顺序、调用哪些 Skill。

**不预设场景，不穷举分类，根据诉求本身去思考。**

**支持两种工作模式：**
- **创建模式**：从零生成全新的 Spec 文档
- **变更模式**：基于已有 Spec 做增量修改

### 三条核心原则

1. **用户确认**：所有资源创建决策都需要用户确认，AI 不自主决策
2. **代码真相源**：每次都扫描代码库获取最新状态，不靠记忆
3. **复用优先**：能复用现有资源就复用，避免重复建设

### 决策规则

```
用户明确说明需要 → 直接生成任务
用户未说明 → AI 分析是否需要 → 输出决策清单 → 用户确认 → 生成任务
```

---

## 工作模式识别（第一步，进入工作流之前）

**进入 Skill 后首先判断用户意图属于哪种模式：**

### 模式判定规则

| 信号 | 模式 | 说明 |
|------|------|------|
| 用户提到"修改 Spec"、"更新 Spec"、"改一下 Spec"、"Spec 里 xxx 改成 yyy" | **变更模式** | 用户明确要修改已有 Spec |
| 用户提到某个已有 Spec 的具体内容（如任务名、决策编号）并要求调整 | **变更模式** | 虽未说"修改 Spec"，但意图是变更 |
| 用户的修改意见指向一个已存在于 `docs/specs/` 的 Spec 文件 | **变更模式** | 存在可修改的目标 |
| 用户要求"生成 Spec"、"写 Spec"、"创建实施计划"且不指向已有 Spec | **创建模式** | 从零开始 |
| 无法判断 | **询问用户** | "您是要修改已有的 xxx-spec.md，还是创建新的 Spec？" |

**判定为创建模式 → 走下方「工作流程」（第0步~第6步）**
**判定为变更模式 → 走下方「变更模式工作流」**

## 可调度的下游 Skill

| Skill | 职责 | 典型产出 |
|-------|------|---------|
| create-route | 页面路由配置（pages.json + 路由常量） | `src/pages.json` 配置 + `src/constants/routes.ts` 常量 |
| create-api | 接入 API 接口 | `src/http/services/` 下的服务文件 |
| create-store | 创建/修改 Pinia Store | `src/store/` 下的 Store 文件 |
| create-hook | 创建/修改 Composable | `src/hooks/` 下的 Hook 文件 |
| create-util | 创建/修改工具函数 | `src/utils/` 下的工具文件 |
| create-sfc | 创建/修改完整的 Vue SFC 文件（template + script + style）。**一个 `.vue` 文件 = 一个 `create-sfc` 任务** | 页面/组件的完整 `.vue` 文件 |
| design-analysis | 分析设计稿，产出 UI 清单。**⚠️ 前置条件：必须有设计稿作为输入源。有设计稿时必须在 create-proposal 之前先执行** | UI 清单文档 |

**AI 根据诉求自由组合这些 Skill，不限定固定的组合方式。**

> **路由配置归属说明**：新建/修改页面路由（`pages.json` + `routes.ts`）统一由 `create-route` 处理。`create-sfc` 不负责路由注册。

> **⚠️ design-analysis 的定位**：`design-analysis` 是 Spec 的**前置步骤**（有设计稿时）。它的产出（UI 清单）是 `create-proposal` 分析组件复用/抽离的精确输入源。没有设计稿 → 禁止调用 `design-analysis`。

## 工作流程（创建模式）

> **以下第0步~第6步为「创建模式」的完整流程。变更模式有独立的工作流，见下方「变更模式工作流」章节。**

```
用户提供诉求
    ↓
第0步：有设计稿？ → 先调用 design-analysis 产出 UI 清单
    ↓
第1步：理解诉求
    ↓
第2步：按需扫描代码库
    ↓
第3步：分析（复用 + 逻辑点提取 + 抽离分类）
    ↓
第4步：输出决策清单（⚠️ 必须暂停，等待用户确认）
    ↓
第5步：生成提案文档
    ↓
第6步：交付确认（⚠️ 必须暂停，create-proposal 到此结束）
    ↓ （用户手动触发）
第7步：执行任务（仅在用户手动触发时执行）
```

---

## 第0步：design-analysis 前置（有设计稿时强制执行）

**如果用户提供了设计稿截图**，必须在进入第1步之前先调用 `design-analysis` Skill 产出 UI 清单文档。

```
有设计稿 → 调用 design-analysis → 产出 UI 清单 → 再进入第1步
无设计稿 → 直接进入第1步
```

**为什么必须前置**：UI 清单是 `create-proposal` 精确分析组件复用/抽离的依据。没有 UI 清单就无法准确判断需要几个 `create-sfc` 任务、每个任务的范围是什么。

**产出物**：`docs/ui-analysis/<名称>-ui-inventory.md`

---

## 第1步：理解诉求

**目的**：搞清楚用户到底要做什么，而不是急着分类。

### 需要想清楚的问题

1. **用户要达成什么目标？**（不是"输入了什么文档"，而是"要什么结果"）
2. **涉及哪些资源类型？**（页面？组件？Hook？Store？API？Util？）
3. **有没有设计稿？** → 有则第0步已产出 UI 清单
4. **涉及几个页面/模块？** → 多页面联动需要先分析共享资源
5. **是新建还是修改？** → 修改需要先读取现有代码

### 关键原则：输入源决定分析边界

- **有 UI 清单**（设计稿经过 design-analysis） → 基于 UI 清单 + 交互稿 + 代码库分析组件复用/抽离
- **无 UI 清单，有交互稿** → 基于交互稿 + 代码库分析组件复用/抽离
- **有交互文档/API文档** → 涉及逻辑层面（Hook/Store/Util 抽离）
- **无交互文档** → 不涉及逻辑层面

**组件复用/抽离分析始终做**（不依赖是否有设计稿），只是精度不同：有 UI 清单时更精确，无 UI 清单时基于交互稿判断。

---

## 第2步：按需扫描代码库

**目的**：只扫描与诉求相关的资源，建立复用基础。

### 扫描原则

**不是每次都全量扫描。** 根据第1步分析出的涉及资源类型，只扫描需要的部分。

**⚠️ 扫描边界：**
- **无交互文档/API文档** → 禁止扫描 `src/hooks/`、`src/utils/`、`src/store/`、`src/http/services/`（逻辑维度）

| 涉及资源 | 需要扫描 | 前提条件 |
|---------|---------|---------|
| API | `src/http/services/` 下相关服务 | 有交互文档/API文档 |
| Store | `src/store/index.ts` | 有交互文档/API文档 |
| Hook | `src/hooks/index.ts` | 有交互文档/API文档 |
| Util | `src/utils/index.ts` | 有交互文档/API文档 |
| 组件 | `src/components/index.ts` | **始终扫描**（组件复用/抽离分析始终需要） |
| 页面 | `src/pages/` 目录结构 + `src/pages.json` | 始终扫描 |
| 路由 | `src/constants/routes.ts` + `src/pages.json` | 始终扫描 |

### 始终扫描

无论什么诉求，以下始终需要扫描（因为可能存在复用机会）：

- `src/pages.json`（路由全局配置）
- `src/constants/routes.ts`（路径常量）
- `src/components/index.ts`（已有组件清单，用于组件复用/抽离分析）

## 第3步：分析（复用 + 逻辑点提取 + 抽离分类）

**目的**：基于扫描结果，分析可复用资源；从输入文档中穷举所有逻辑点；对每个逻辑点做确定性分类。

### 3.1 复用分析

对于诉求涉及的每种资源，检查现有代码库是否已有可复用的：

- **完全匹配** → 直接复用
- **部分匹配** → 评估是否可扩展现有资源
- **无匹配** → 需要新建

组件复用时，遵循 `04-组件规范` 的优先级：本地组件 > TDesign 组件 > 新建。

### 3.2 逻辑点提取（穷举，不遗漏）

**核心原则：按维度框架逐一扫描输入文档，提取所有独立的逻辑点。不靠"想到什么算什么"，而是按框架逐条过。**

扫描维度：
- **逻辑维度**：有交互文档/API文档时扫描
- **UI 维度（组件复用/抽离）**：**始终扫描**——有 UI 清单时基于 UI 清单 + 交互稿 + 代码库分析；无 UI 清单时基于交互稿 + 代码库分析

#### 逻辑维度扫描框架（有交互文档时使用）

| 扫描维度 | 要识别的逻辑点 | 示例 |
|---------|--------------|------|
| **表单字段** | 每个字段的校验规则（逐字段扫描） | 手机号校验规则、密码强度规则、邮箱格式规则 |
| **UI 交互行为** | 切换、展开/折叠、显隐、拖拽等有状态的交互 | 密码明文/密文切换、折叠面板、Tab 切换 |
| **异步操作** | API 调用、定时器、轮询、倒计时等 | 登录请求、验证码倒计时、轮询审批状态 |
| **状态管理** | **必须逐个分析每个 API 的返回值类型**，对返回值中的每个字段判断：①仅当前页面一次性消费？②会被其他页面/模块读取（跨页面共享）？③需要在 app 生命周期内持久保留？只要存在②或③的字段，就需要全局状态管理。同时扫描交互文档中是否有"需要记住/保持/跨页面使用"的数据描述 | 认证凭证、用户档案、全局配置、购物车、权限角色等 |
| **导航流程** | 页面跳转、重定向、返回、拦截等 | 登录成功跳首页、未登录拦截跳登录页 |

#### UI 维度扫描框架（始终执行，精度取决于输入源）

**有 UI 清单** → 基于 UI 清单 + 交互稿 + 代码库分析，精度高
**无 UI 清单** → 基于交互稿 + 代码库分析，精度较低但仍能识别明显的组件抽离需求

| 扫描维度 | 要识别的组件抽离点 | 示例 |
|---------|------------------|------|
| **重复 UI 结构** | 在同一页面或跨页面中出现 2+ 次的相同/相似 UI 模块 | 列表项卡片、操作按钮组、空状态提示 |
| **复杂独立区块** | 内部结构复杂（3+ 层嵌套或含独立交互）的 UI 区域 | 富文本编辑器、图片上传区、评论区 |
| **带状态的 UI 片段** | 有自身交互状态的 UI 区域（展开/收起、选中态、动画等） | 手风琴面板、可选标签组、搜索筛选栏 |
| **业务卡片/单元** | 展示一组业务数据的独立 UI 单元 | 用户信息卡片、订单卡片、审批记录条目 |
| **弹窗/浮层** | 所有模态弹窗、底部弹出、Toast、操作菜单等 | 确认弹窗、选择器弹窗、操作菜单 |

**要求：必须把每个识别出的逻辑点逐条列出，形成"逻辑点清单"，作为下一步分类的输入。**

### 3.3 抽离分类（确定性规则，不靠猜测）

**对 3.2 提取出的每个逻辑点，按以下规则链顺序判断，命中第一条即停止：**

```
逻辑点
  │
  ├─ 代码库中已有【精确匹配】的可复用资源？
  │   └─ YES → 🟢 直接复用（不问用户）
  │
  ├─ 是纯粹的业务编排流程？（见下方严格定义）
  │   └─ YES → 🟢 页面内实现（不问用户）
  │
  └─ 其余所有
      └─ 🔶 列入抽离决策（问用户）
```

#### 规则一："直接复用"的严格判定

**只有"精确匹配"才算复用，"框架匹配"不算。**

```
精确匹配 = 代码库中已有资源直接提供了该逻辑点的完整功能，无需编写新代码
框架匹配 = 代码库中有处理同类问题的框架/容器，但具体逻辑仍需新写
```

| 情况 | 判定 | 说明 |
|------|------|------|
| 已有 `useFormValidation` hook，本次需要手机号校验规则 | ❌ 不算复用 | hook 是校验框架，手机号正则是具体逻辑，需要新写 → 🔶 问用户 |
| 已有 `phoneValidator` util，本次需要手机号校验 | ✅ 精确复用 | 功能完全匹配 → 🟢 直接复用 |
| 已有 `router` util，本次需要跳转首页 | ✅ 精确复用 | 调用 `router.navigateTo()` 即可 → 🟢 直接复用 |
| 已有 `useLoading` hook，本次需要加载态 | ✅ 精确复用 | 功能完全匹配 → 🟢 直接复用 |

**判断口诀：问自己"这个逻辑点是否需要写新的函数/正则/规则/配置？" — 需要写 → 不算复用。**

#### 规则二："页面内实现"的严格定义

**只有"纯粹的业务编排流程"才归类为页面内实现。**

```
纯粹的业务编排 = 把多个已有资源串联起来的流程代码，本身不包含可独立的逻辑单元
```

✅ 符合"页面内实现"的例子：
- "点击提交按钮 → 调用校验 → 调 API → 存储返回数据 → 跳转下一页" — 这是串联流程
- "页面 onLoad 时读取路由参数并设置页面标题" — 这是页面初始化编排

❌ 不符合"页面内实现"的（应列为决策点）：
- 密码明文/密文切换 — 有独立的响应式状态（ref），可脱离当前页面独立存在
- 手机号校验正则 — 纯函数，可脱离当前页面独立存在
- 倒计时逻辑 — 有独立的状态和副作用，可脱离当前页面独立存在

**判断口诀：问自己"这段逻辑能否脱离当前页面的业务上下文独立存在？" — 能 → 不算页面内实现 → 🔶 问用户。**

#### 规则三：其余所有 → 问用户

不符合以上两条的，**一律**列入抽离决策。AI 附上建议的资源类型：
- **有响应式状态（ref/reactive）** → 建议 Hook
- **纯函数、无副作用** → 建议 Util
- **全局状态、跨页面共享** → 建议 Store
- **UI 结构复杂、可复用** → 建议组件

### 3.4 接口字段覆盖度检查（有 API 调用时强制执行）

**当诉求涉及 API 调用时，必须做双向检查：入参覆盖度 + 返回值数据流向。两者缺一不可。**

#### 3.4.1 入参覆盖度检查

**"API 入参字段 vs 交互文档表单字段"的逐一映射，识别缺口。**

##### 执行步骤

1. **列出 API 入参字段**：从 API 类型定义中提取所有字段（必填+选填）
2. **列出交互文档表单字段**：从交互文档中提取用户需要填写的所有表单字段
3. **逐一映射**：把 API 字段和交互文档字段一一对应
4. **识别缺口**：找出以下两类问题
   - **API 需要但交互文档未设计**：API 入参中有字段，但交互文档中没有对应的 UI 输入
   - **交互文档有但 API 不需要**：交互文档中有字段，但 API 入参中没有（如"确认密码"仅前端校验）

##### 缺口处理

| 缺口类型 | 处理方式 |
|---------|---------|
| API 需要但交互文档未设计 | 列为 🔶 决策点，问用户：是传默认值还是需要补充 UI 输入 |
| 交互文档有但 API 不需要 | 标注为"仅前端使用"，在 Spec 中注明不提交给 API |

##### 输出格式

```
入参覆盖度检查：
| API 字段 | 交互文档字段 | 状态 |
|---------|------------|------|
| userName | 企业账户名 | ✅ 匹配 |
| invitationCode | — | ⚠️ API 需要但交互文档未设计 |
| — | confirmPassword | ℹ️ 仅前端校验 |
```

#### 3.4.2 返回值数据流向检查

**必须读取每个 API 的返回值类型定义（Response Type），逐字段分析数据在应用中的消费方式，自主判断是否需要全局状态管理。**

##### 执行步骤

1. **读取 API 返回值类型定义**：找到 Response 类型（包括嵌套类型），列出所有返回字段及其语义
2. **逐字段做三个判断**：

   | 判断问题 | YES 的含义 |
   |---------|-----------|
   | 这个数据只在当前页面用一次就丢弃？（如操作结果提示、一次性列表） | → 页面内消费，不需要 Store |
   | 这个数据会被**其他页面/组件/拦截器**读取或使用？ | → 跨模块共享 → 需要 Store |
   | 这个数据在用户关闭页面/重启应用后仍需保留？ | → 持久化 → Store + 持久化策略 |

3. **识别 Store 需求**：只要某个返回字段命中了"跨模块共享"或"持久化"中的任意一条
   - 检查代码库中是否**已有对应 Store** 能存储该数据
   - 已有 → 🟢 直接复用
   - 没有 → 🔶 列为决策点（建议 Store），附上 AI 的判断理由

##### 判断指引（帮助 AI 自主分析，非穷举）

以下是一些常见的**信号特征**，帮助判断返回值是否可能需要全局存储。但最终判断应基于对该数据在应用中实际消费场景的分析，而非机械匹配关键词：

- 返回值中包含**凭证/令牌类**字段 → 通常需要跨模块（如请求拦截器）使用
- 返回值中包含**实体档案类**字段（用户、组织、角色等） → 通常多个页面需要展示
- 返回值中包含**配置/偏好类**字段 → 通常影响全局行为
- 返回值中包含**权限/能力类**字段 → 通常多处需要做权限判断

##### 输出格式

```
返回值数据流向检查：
| API | 返回字段 | 语义分析 | 判断结论 |
|-----|---------|---------|---------|
| xxxApi | fieldA | （AI 自主分析该字段的用途和消费场景） | 页面内消费 / ⚠️ 需要全局存储（理由） |
| xxxApi | fieldB | （AI 自主分析该字段的用途和消费场景） | 页面内消费 / ⚠️ 需要全局存储（理由） |
```

**核心原则：每次涉及 API 调用的 Spec，入参覆盖度和返回值数据流向两项检查都必须做。不可遗漏。**

### 3.5 隐含假设识别（强制执行）

**核心原则：当你写出一条任务需求时，审视其中每个功能点的交互行为。如果某个行为在交互文档/设计稿中没有被明确描述（原文找不到对应依据），则该行为属于「隐含假设」，必须提取为决策项让用户确认。**

这不是一个固定的检查清单，而是一条**通用的元规则**——无论遇到什么新场景都适用。

#### 执行方式

对每个任务中的每个功能点，问自己：

> **这个行为的依据是什么？是需求文档/设计稿中明确描述的，还是我自己推断补充的？**

- **有明确依据**（能指出文档/设计稿中的具体描述） → ✅ 直接写入任务需求
- **没有明确依据**（是 AI 基于常见做法推断的） → 🔶 必须作为决策项问用户

#### 帮助理解的示例（不是穷举清单）

- 交互文档说「校验密码格式」，但没说**什么时候校验**（失焦？提交时？混合？） → 🔶 决策
- 交互文档说「注册成功后跳转」，但没说**用 navigateTo 还是 redirectTo** → 🔶 决策
- 交互文档说「显示错误」，但没说**用 inline 文字还是 toast** → 🔶 决策
- 交互文档说「发送验证码」，但没说**倒计时秒数是多少** → 🔶 决策
- 设计稿中有表单，但没说**空表单时提交按钮是否禁用** → 🔶 决策

**本质：AI 不应该对任何用户未明确说明的交互行为做默认假设。如果你发现自己在"凭经验补全"一个行为细节，那它就是一个隐含假设，必须问用户。**

### 3.6 分类结果自检（强制执行）

**分类完成后，必须逐条执行以下自检，不可跳过：**

```
对每个被标记为 🟢 的逻辑点，反问自己：

□ 标记为"直接复用"的：代码库中是否真的有精确匹配的资源？
  还是只是有一个同类框架？（框架 ≠ 精确匹配）

□ 标记为"页面内实现"的：这段逻辑是否包含可独立存在的函数/状态/规则？
  还是真的只是串联已有资源的流程代码？

□ 状态管理维度是否已扫描？是否逐个读取了每个 API 的返回值类型定义，
  并对每个返回字段做了"跨模块共享？持久化？"的判断？
  是否存在漏掉的返回字段未被分析？

□ 3.4 接口检查是否做了双向检查（入参覆盖度 + 返回值数据流向）？
  是否只检查了入参而遗漏了返回值分析？

□ 3.5 隐含假设识别是否已执行？是否存在 AI 自行补全的交互行为细节
  没有被提取为决策项？

如果自检发现判断有误 → 改为 🔶 问用户
```

**核心原则：宁可多问，不可漏问。AI 绝对不要替用户判断"未来会不会复用"，也不要替用户决定交互行为的细节。有疑问就问。**

### 3.7 SFC 任务的 Skill 选择约束（强制执行）

**所有涉及 `.vue` 文件创建或修改的任务，统一使用 `create-sfc` Skill。**

**一个 `.vue` 文件 = 一个 `create-sfc` 任务。** `create-sfc` 会根据是否有 UI 清单自动决定行为模式：

| 场景 | create-sfc 行为 |
|------|----------------|
| **有 UI 清单** | 创建完整 SFC：template 精确还原设计稿 + 完整 script 业务逻辑 + 精确 style |
| **无 UI 清单，有交互稿** | 创建完整 SFC：template 按交互稿 + 组件规范铺基础 UI + 完整 script 业务逻辑 + 基础合理 style |

**⚠️ 关键规则：**

1. **禁止将一个 `.vue` 文件的 template/style 和 script 拆成两个任务**（那是旧方案的做法）
2. **如果需要抽离组件**（由 UI 维度分析发现），每个组件 = 一个独立的 `create-sfc` 任务
3. **`design-analysis` 不在 Spec 的任务清单中**，它是 Spec 的前置步骤（第0步已完成）
4. **设计稿后来了** → 再走一轮 `design-analysis` + `create-sfc`（修改模式），用 UI 清单做视觉还原

**判断口诀：看到 `.vue` 文件 → 用 `create-sfc`。一个文件一个任务，不拆不合。**

## 第4步：输出决策清单（⚠️ 必须暂停）

**这是最关键的步骤！AI 必须在此步骤暂停，等待用户确认所有决策。**

### 决策清单内容

根据诉求实际情况，输出以下内容（**只输出与诉求相关的部分，不相关的不要出现**）：

#### 一、任务拆分方案

列出 AI 分析后认为需要执行的任务。**每个任务必须标注执行方式**（Skill 名称 或 "直接执行"）。

**执行方式判断规则：**

**核心原则：如果一个任务的工作内容落在某个可用 Skill 的职责范围内，就必须标注该 Skill，不允许标为"直接执行"。**

| 判断条件 | 执行方式 | 示例 |
|---------|---------|------|
| 任务涉及的工作有对应的可用 Skill | 标注对应 Skill 名称 | 新建页面 → `create-sfc`；新建路由 → `create-route`；新建 Hook → `create-hook` |
| 任务涉及的工作**没有**任何可用 Skill 能覆盖 | 标注为"直接执行" | 修改一个常量值、往已有数组中加一行、改个配置项等 |

**判断口诀：先查"可调度的下游 Skill"列表，看任务的工作内容是否匹配某个 Skill 的职责。匹配 → 用 Skill；不匹配 → 直接执行。**

> ⚠️ 不要因为任务"看起来简单"就标为"直接执行"。即使只是新增一个简单的 Hook 文件，也应该调用 `create-hook` Skill，因为 Skill 内部有文件组织、命名约定、导出注册等规则需要遵守。

#### 二、页面路由决策（涉及新建页面时必须输出）

**每次涉及新建页面，都必须自动生成此决策点：**

> 🔶 **决策：页面分包归属**
>
> `<页面名称>` 放在哪里？
> - A）主包（`pages/<name>/index`）
> - B）已有分包 — 列出当前 `pages.json` 中 `subPackages` 的所有分包名称供选择（如无分包则不列此项）
> - C）新建分包（需指定分包名称，kebab-case）

**决策结果影响：**

| 选择 | 路由常量 | 页面路径 | pages.json 配置位置 |
|------|---------|---------|-------------------|
| A 主包 | `PAGE_<NAME>` | `/pages/<name>/index` | `pages` 数组 |
| B 已有分包 | `PAGE_SUB_<NAME>` | `/pages-sub/<module>/<name>/index` | 对应 `subPackages` 条目 |
| C 新建分包 | `PAGE_SUB_<NAME>` | `/pages-sub/<module>/<name>/index` | 新增 `subPackages` 条目 |

> 同时确认导航栏配置：有设计稿时从设计稿提取（标题文字、字体颜色、背景色、是否自定义导航栏），无设计稿时默认仅配置标题文字。

#### 三、资源复用决策

列出诉求涉及的资源需求，以及现有代码库中是否有可复用的。

#### 四、抽离决策

对于用户未明确说明、但 AI 分析认为有抽离价值的资源，列出决策问题让用户确认。

#### 五、交互行为决策（由 3.5 隐含假设识别产出）

列出 AI 在分析过程中发现的「隐含假设」——即需求文档/设计稿中未明确描述、但实现时必须确定的交互行为细节。每个决策项给出可选方案供用户选择。

> ⚠️ 此类决策可能出现在任何场景中，不限于以下示例。核心判断标准是：**这个行为在文档中有没有明确描述？没有就问。**

#### 六、已确定的信息

列出可以确定的信息（已有的 API、已有的 Hook/Util/Store 等）。

### 绝对禁止

- ❌ 跳过此步直接输出定稿提案
- ❌ AI 自行决定任务拆分方案不问用户
- ❌ 把 AI 的建议直接当确定方案写入提案
- ❌ **将一个 `.vue` 文件的 template/style 和 script 拆成两个任务**（统一使用 `create-sfc`）
- ❌ **没有设计稿时拆出 `design-analysis` 任务**（design-analysis 必须有设计稿）
- ❌ **没有交互文档/API文档时拆出逻辑层面任务**（如 `create-store`、`create-hook`、`create-util`、`create-sfc`）

### 用户回复选项

```
1. 💬 逐条回复 → 根据回答生成定稿提案
2. ⏭️ 全部按建议来 → 按建议列填入提案
3. 🔀 部分回复，部分待补充
4. ✏️ 整体调整
```

**等待用户回复后，进入下一步。**

## 第5步：生成提案文档

### 文档保存位置

```
docs/specs/<name>-spec.md
```

### 文档结构

提案文档包含以下部分（**根据诉求裁剪，不相关的章节不写**）：

1. **需求摘要** — 用户要做什么、涉及哪些页面/模块
2. **任务清单** — 所有任务按依赖排序，每个任务标注执行方式（Skill 调用 / 直接执行）
3. **任务详情** — 每个任务的需求描述、使用的已有资源、预期产出
4. **资源方案总览** — 涉及的组件/Store/Hook/Util/API 汇总
5. **决策记录** — 用户确认的所有决策
6. **执行顺序** — 任务执行的先后顺序
7. **变更记录** — 文档变更历史

### ⛔ 输出禁令（强制执行，违反即失败）

Spec 文档中 **绝对禁止** 出现以下内容，无论任何理由：

1. **代码块**（` ``` `，任何语言的围栏代码块）
2. **函数签名**（如 `(value: string) => boolean`、`function xxx(): void`）
3. **TS/JS 接口或类型定义**（如 `interface XxxReturn { ... }`、`type Xxx = ...`）
4. **正则表达式**（如 `/^1[3-9]\d{9}$/`）
5. **具体的 state/actions/getters 字段定义表**
6. **导入语句**（如 `import { xxx } from '@/xxx'`）
7. **具体的函数参数名和返回值类型定义**

以上全部属于 **"怎么做"**，由下游 Skill 根据需求描述和项目规范自行决定。

**⛔ 自检方法**：生成 Spec 后，全文搜索 ` ``` `。如果存在任何围栏代码块 → **必须删除**，改为自然语言描述。如果存在函数签名、接口定义、正则 → **必须删除**，改为自然语言描述需求。

**⛔ 正确做法 vs 错误做法**：

| ❌ 错误（包含实现细节） | ✅ 正确（只描述需求） |
|----------------------|---------------------|
| 创建 `validatePhone(value: string): true \| string` | 创建手机号格式校验函数 |
| 正则 `/^1[3-9]\d{9}$/` 校验手机号 | 校验中国大陆 11 位手机号格式 |
| `interface UseAuthCodeReturn { countdown: Ref<number> }` | 创建验证码倒计时 Hook，需要暴露倒计时秒数 |
| Store 的 `state: { token: string, userInfo: UserProfile }` | Store 需要存储登录凭证和用户信息 |

### 产出一致性约束（强制执行）

为确保多次生成的 Spec 在命名、拆分粒度、文件结构上保持一致，以下规则强制执行：

#### 1. 命名必须参考现有资源

新建资源的命名 **必须先扫描代码库中同类资源的命名风格**，保持一致：
- 新建 Store → 先扫描 `src/store/` 下已有文件名，参考其命名风格（如已有 `auth.ts` 则新 Store 也用功能域命名）
- 新建 Hook → 先扫描 `src/hooks/` 下已有文件名
- 新建 Util → 先扫描 `src/utils/` 下已有文件名

**禁止**凭语感随意命名。如代码库中暂无同类资源，则严格遵循项目编码规范中的命名规则。

#### 2. 任务拆分与执行方式标注

每个任务必须标注执行方式。判断规则：**有匹配的可用 Skill → 标注 Skill 名称；没有匹配的 Skill → 标注"直接执行"。**

**Skill 任务约束：禁止**将不同 Skill 范畴的工作合并到同一任务中：
- 路由配置（`create-route` 范畴）不应合并到 SFC 任务（`create-sfc` 范畴）中
- Store 创建（`create-store`）不应与 Hook 创建（`create-hook`）合并
- 路由常量和 pages.json 配置属于同一 Skill（`create-route`），应合并为一个任务

**路由配置任务归属：**
- 新增/修改 `src/constants/routes.ts` 路由常量 + `src/pages.json` 页面路由配置 → **统一由 `create-route` 处理**
- 禁止将路由常量修改分配给 `create-util`（路由常量不是工具函数）
- 禁止将 pages.json 修改分配给 `create-sfc`（pages.json 不是 SFC 文件）

**判断口诀：如果一个任务需要修改不同 Skill 职责范围的文件，则应拆为独立任务。但路由常量和 pages.json 同属路由配置范畴，必须合并到一个 `create-route` 任务。**

**直接执行任务约束：** 只有当任务的工作内容**不属于任何可用 Skill 的职责范围**时，才标注为"直接执行"。典型场景：修改一个已有常量的值、往已有配置对象/数组中追加一行等。

#### 3. 文件结构必须遵循项目规范

**在 Spec 中指定产出文件路径时，必须以对应 Rule 文件中的文件组织规定为准：**
- 路由配置 → 以 `06-路由规范.mdc` 中的规定为准（路径常量在 `src/constants/routes.ts`，路由配置在 `src/pages.json`）
- Hook 文件结构 → 以 `10-Hooks规范.mdc` 中的规定为准
- Util 文件结构 → 以 `09-工具函数规范.mdc` 中的规定为准
- Store 文件结构 → 以 `07-状态管理.mdc` 中的规定为准

**禁止**在 Spec 中自行发明文件组织方式（如规范要求单文件 `use-xxx.ts`，则不能写成目录形式 `use-xxx/index.ts`）。

#### 4. 决策记录完整度

决策记录章节必须记录以下两类决策：
- **显式决策**：用户在决策清单中明确回答的选择
- **隐含决策**：分析过程中 AI 判断为"页面内实现"（不抽离）的逻辑点，也需记录为决策（用户确认不抽离 = 一个决策）

### 任务排序原则

根据任务之间的实际依赖关系排序：
- **被依赖的任务排前面**（如 Store 被页面逻辑依赖，则 Store 先建）
- **无依赖关系的任务可以并行**
- **不预设固定的 Skill 顺序**，根据实际情况决定

### 文档状态

| 标记 | 含义 |
|------|------|
| `🔶 草稿` | 正在确认决策中 |
| `🟢 定稿` | 所有决策已确认，可以执行 |
| `🔵 执行中` | 任务正在逐个执行 |
| `✅ 已完成` | 所有任务执行完毕 |

向用户展示定稿摘要，包含：
- Spec 文件路径
- 任务数量
- 待补充信息（如有）

**⚠️⚠️⚠️ 核心约束：Spec 生成后 create-proposal 的工作即告结束。绝对禁止自动进入第7步执行任务。必须完全停止，等待用户手动触发执行。**

## 变更模式工作流

> **以下为「变更模式」的专用流程。当模式识别判定为变更模式时进入此流程。**

### 变更模式总览

```
用户提出修改意见
    ↓
V1：定位目标 Spec
    ↓
V2：读取现有 Spec + 解析已有决策和任务
    ↓
V3：理解修改意见 + 变更影响分析
    ↓
V4：输出变更决策清单（⚠️ 必须暂停，等待用户确认）
    ↓
V5：增量更新 Spec 文档
    ↓
V6：交付确认（⚠️ 必须暂停，create-proposal 到此结束）
```

---

### V1：定位目标 Spec

**目的**：确定要修改的是哪个 Spec 文件。

1. **用户明确指定**（如"修改 register-spec"）→ 直接定位 `docs/specs/register-spec.md`
2. **用户未指定但上下文可推断**（如上一轮对话刚生成了某个 Spec）→ 推断目标
3. **无法确定** → 扫描 `docs/specs/` 目录列出所有活跃 Spec 文件，请用户选择

**确认目标 Spec 文件存在后进入下一步。文件不存在则提示用户。**

---

### V2：读取现有 Spec + 解析

**目的**：完整读取目标 Spec，建立"当前状态快照"，作为增量修改的基础。

**必须提取的信息：**

| 提取项 | 来源章节 | 用途 |
|--------|---------|------|
| 所有任务及其依赖关系 | 二、任务清单 + 三、任务详情 | 判断修改是否影响现有任务 |
| 所有已确认决策 | 六、决策记录 | 保留未被修改的决策 |
| 资源方案总览 | 五、资源方案总览 | 判断新增/删除资源的影响 |
| 交互要点 | 一、需求摘要 → 交互要点 | 判断修改是否影响交互描述 |
| 当前版本号 | 变更记录 | 确定新版本号 |
| 关联文档 | 文档头部元信息 | 保留关联信息 |

**⚠️ 关键：必须完整读取，不能只读用户提到的章节。因为一个修改可能级联影响多个章节。**

### V3：理解修改意见 + 变更影响分析

**目的**：搞清楚用户要改什么，以及这个修改会级联影响 Spec 的哪些部分。

#### 3.1 修改意见分类

将用户的修改意见逐条归类：

| 修改类型 | 示例 | 影响范围 |
|---------|------|---------|
| **决策变更** | "D7 改成抽离组件" | 决策记录 + 可能新增/删除任务 + 资源总览 + 任务详情 + 执行顺序 |
| **任务调整** | "T2 增加一个校验函数" | 任务详情 + 可能影响资源总览 |
| **新增需求** | "加一个忘记密码的跳转链接" | 需要走「逻辑点提取 → 抽离分类」分析，可能新增决策 + 任务 |
| **删除需求** | "不需要邮箱注册了，只支持手机号" | 任务详情修改 + 可能删除任务 + 资源总览 + 决策记录 |
| **交互变更** | "校验改成失焦+提交混合模式" | 交互要点 + 相关任务详情 |
| **参数调整** | "倒计时改成 120 秒" | 决策记录 + 相关任务详情 |

#### 3.2 变更影响分析（核心步骤）

**对每条修改意见，逐一分析其影响的 Spec 章节：**

```
修改意见
  │
  ├─ 影响「需求摘要 → 交互要点」？ → 需要更新交互要点列表
  │
  ├─ 影响「任务清单」？
  │   ├─ 需要新增任务？ → 确定新任务的 Skill、依赖关系
  │   ├─ 需要删除任务？ → 检查其他任务是否依赖被删任务
  │   └─ 需要修改任务依赖？ → 更新依赖链
  │
  ├─ 影响「任务详情」？ → 哪些任务的需求描述需要更新
  │
  ├─ 影响「资源方案总览」？ → 哪些资源需要新增/删除/修改
  │
  ├─ 影响「决策记录」？ → 哪些决策需要更新
  │
  └─ 影响「执行顺序」？ → 是否需要调整任务执行顺序
```

#### 3.3 新增需求的完整分析

**如果修改意见引入了新的功能需求（不是简单参数调整），必须对新需求走创建模式中的分析流程：**

1. **按需扫描代码库**（同创建模式第2步，但只扫描与新需求相关的部分）
2. **逻辑点提取**（同创建模式第3.2步）
3. **抽离分类**（同创建模式第3.3步规则链）
4. **隐含假设识别**（同创建模式第3.5步）

**这确保新增需求的分析质量与创建模式完全一致。**

#### 3.4 变更影响自检（强制执行）

```
对每条修改意见，检查：

□ 是否遗漏了级联影响？（如删除一个 Hook → 使用该 Hook 的 SFC 任务也需更新）
□ 新增任务是否正确选择了 Skill？（同创建模式 3.7 的 Skill 选择约束）
□ 任务依赖关系是否仍然正确？（新增/删除任务后依赖链是否需要调整）
□ 资源总览是否与任务清单一致？（任务中涉及的资源是否都在总览中）
□ 如果涉及新增需求，是否走了完整的逻辑点提取+分类流程？
□ 执行顺序是否需要更新？
```

### V4：输出变更决策清单（⚠️ 必须暂停）

**与创建模式一样，所有需要用户确认的变更都必须在此步暂停，等用户确认。**

#### 变更决策清单格式

输出以下结构：

**一、变更摘要**

> 简述用户的修改意见和整体影响。

**二、受影响的章节**

列出哪些章节会被修改，以及具体变更内容预告。

**三、需要用户确认的新决策**（如有）

> 仅当修改意见引入了新的决策点时才输出此部分。

| # | 决策问题 | AI 建议 | 选项 |
|---|---------|--------|------|
| D(N+1) | ... | ... | A / B / ... |

**四、无需确认的变更**

> 列出用户已明确指示的变更内容（如"倒计时改成 120 秒"），这些不需要再确认，但需展示给用户知悉。

**五、变更前后对比**（重要变更）

> 对于影响较大的变更（如新增/删除任务），展示变更前后的对比。

#### 何时不需要新决策

如果用户的修改意见足够明确且不引入新的不确定性，则可以没有"需要用户确认的新决策"部分。此时仍需展示变更摘要和受影响章节，让用户确认后再更新。

#### 用户回复选项

```
1. 💬 逐条确认 → 根据回答更新 Spec
2. ⏭️ 全部确认 → 按展示的内容更新 Spec
3. 🔀 部分调整
```

**等待用户回复后，进入下一步。**

### V5：增量更新 Spec 文档

**目的**：根据用户确认的变更，精确更新 Spec 文件。

#### 更新原则

1. **只改需要改的**：未受影响的章节保持原样，不做任何修改
2. **保持格式一致**：更新后的内容必须严格遵循 `output-spec.md` 模板格式
3. **保留已有决策**：未被修改的决策保持原记录不变
4. **追加变更记录**：在「变更记录」章节追加新版本

#### 必须更新的章节

| 章节 | 更新条件 |
|------|---------|
| 需求摘要 → 交互要点 | 修改影响交互行为时 |
| 任务清单 | 新增/删除/修改任务时 |
| 任务详情 | 任务需求描述变化时 |
| API 接口清单 | API 使用变化时 |
| 资源方案总览 | 资源新增/删除时 |
| 决策记录 | 有新决策或决策变更时 |
| 执行顺序 | 任务清单变化时 |
| 变更记录 | **每次变更必须追加** |

#### 任务编号规则

- **新增任务**：使用连续编号（如已有 T1~T6，新增为 T7）
- **删除任务**：删除该任务，后续任务**不重新编号**（保持引用稳定性）
- **如果删除导致编号出现大量间隙**（超过 2 个连续缺失），可以在变更记录中说明，但仍不重编号

#### 决策编号规则

- **新决策**：在已有最大编号基础上递增（如已有 D1~D17，新增为 D18）
- **决策变更**：保留原编号，更新决策内容，在"用户决定"列注明"**（变更）**"+ 新决定

#### 变更记录格式

```
### v<N> — <变更概述>（YYYY-MM-DD）
- 变更原因：<用户的修改意见概述>
- 具体变更：
  - [章节名] 变更描述
  - [章节名] 变更描述
- 受影响任务：T?、T?
```

#### ⛔ 输出合规性检查（与创建模式相同）

更新后的 Spec 仍需遵守创建模式的所有输出禁令：不包含代码块、函数签名、TS 接口/类型定义、正则表达式、import 语句等。

## 提案文档管理

### 代码作为真相源

每次生成提案都全新分析代码库，不依赖历史提案文档。

### 归档

- **活跃文档**：`docs/specs/<name>-spec.md`
- **归档文档**：`docs/specs/.archive/<name>/<version>-tasks.md`
- **触发**：手动归档，或活跃文档中已完成任务超过 5 个时自动归档

## 职责边界

**create-proposal 只说"做什么"，下游 Skill 决定"怎么做"。**

- ✅ 分析诉求，识别需要哪些资源
- ✅ 输出决策清单，等待用户确认
- ✅ 编排任务，确定执行顺序
- ✅ 用自然语言描述每个任务的需求
- ❌ 不写代码块、函数签名、接口定义、正则表达式
- ❌ 不教下游 Skill 怎么实现
- ❌ 不替下游 Skill 做抽离以外的技术决策
- ❌ 不指定具体的参数名、返回值类型、字段名
