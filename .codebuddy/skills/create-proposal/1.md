# 智能提案编排系统

## 核心定位

**create-proposal 是整个开发流程的"大脑"**，负责：
1. **判断**：分析需求，识别需要创建的资源
2. **决策**：输出决策清单，让用户确认所有关键决策
3. **编排**：动态组合任务，按正确顺序执行

**关键原则**：
- AI不做自主决策，所有资源创建都需要用户确认
- 代码作为真相源，每次生成提案都全新分析代码库
- 能复用就复用，优先使用现有资源

---

## 统一决策原则

### 核心规则：所有资源创建都需要用户确认

```
用户明确说明需要 → 直接生成任务
用户未说明 → AI分析是否需要 → 输出决策清单 → 用户确认 → 生成任务
```

### 适用范围
- **组件**：用户没说新建 → AI分析抽离机会 → 决策清单
- **Hook**：用户没说新建 → AI分析抽离机会 → 决策清单
- **Util**：用户没说新建 → AI分析抽离机会 → 决策清单
- **Store**：用户没说新建 → AI分析是否需要 → 决策清单

## 工作流程

### 流程总览

```
用户提供输入
    ↓
第0步：输入分析
    ↓
第1步：代码库扫描
    ↓
第2步：复用分析
    ↓
第3步：抽离决策分析
    ↓
第4步：输出决策清单（⚠️ 必须暂停，等待用户确认）
    ↓
第5步：任务编排
    ↓
第6步：生成提案文档
    ↓
第7步：交付确认（⚠️ 必须暂停，等待用户确认）
    ↓
第8步：执行任务
```

## 第0步：输入分析

识别用户提供的输入源类型：

### 输入源类型

| 输入源 | 处理方式 |
|--------|---------|
| 设计稿图片/链接 | 触发 `design-analysis` 技能 → 生成 UI 清单 |
| API 文档 | 直接解析接口信息 |
| 交互文档 | 直接解析交互逻辑 |
| 口述需求 | 直接分析需求内容 |

### 多输入源组合

用户可能同时提供多个输入源，需要综合分析：
- 设计稿 + API 文档 + 交互文档 → 完整场景
- 仅 API 文档 + 交互文档 → 后端驱动场景
- 仅设计稿 → UI 驱动场景
- 仅口述需求 → 快速开发场景

### 操作步骤

1. 识别用户提供的输入源类型
2. 如果有设计稿，先调用 `design-analysis` 技能生成 UI 清单
3. 整理所有输入信息，进入下一步

## 第2步：复用分析

**目的**：优先复用现有资源，避免重复建设

### 组件复用优先级

遵循 `04-组件规范`，优先级从高到低：

1. **本地通用组件** (`src/components/`)
   - 直接使用现有组件
   
2. **TDesign 组件**
   - 使用 `tdesign-uniapp` 技能查阅组件文档
   - 确认 Props/Events/Slots 后使用
   
3. **新建组件**
   - 以上都不满足时才新建

### Hook/Util/Store 复用

| 资源类型 | 匹配规则 | 决策 |
|---------|---------|------|
| Hook | 功能相似 + API 相似 | 复用 / 扩展现有 / 新建 |
| Util | 功能相同 + 参数兼容 | 复用 / 新建 |
| Store | 状态相似 + 用途相似 | 复用 / 新建 |

### 复用分析流程

```
需求 → 扫描现有资源 → 匹配分析 → 决策输出

示例：
需求：密码输入框（带显隐切换）
扫描：无密码输入组件
匹配：无完全匹配
决策：新建组件
```

## 第3步：抽离决策分析

**目的**：识别可抽离的资源，避免代码冗余

### 组件抽离判断

#### 判断依据

1. **复杂度分析**
   - 结构层数 ≥ 3 层
   - 交互数量 ≥ 2 个（如显隐切换 + 输入校验）
   
2. **通用性判断**
   - 符合常见组件特征（按钮/输入框/卡片/列表项）
   - 业务逻辑独立
   
3. **复用预测**
   - 未来可能在其他页面使用
   - 项目中有类似需求场景

#### 决策规则

| 情况 | 判断依据 | 建议 |
|------|---------|------|
| 高复杂度 + 高通用性 | 结构复杂 + 符合组件特征 | ✅ 建议抽离 |
| 高复杂度 + 低通用性 | 结构复杂 + 业务强耦合 | ❌ 不建议抽离 |
| 低复杂度 + 高通用性 | 结构简单 + 可复用 | ⚠️ 可选抽离 |
| 低复杂度 + 低通用性 | 结构简单 + 单次使用 | ❌ 不抽离 |

### Hook 抽离判断

#### 判断依据

1. **多页面共享**
   - 逻辑在 2 个及以上页面使用
   - 有状态管理需求
   
2. **复杂状态管理**
   - 状态逻辑复杂（如多步骤表单）
   - 需要响应式数据

#### 决策规则

| 情况 | 建议 |
|------|------|
| 多页面共享 + 有状态 | ✅ 建议抽离 |
| 单页面 + 复杂状态 | ⚠️ 可选抽离 |
| 单页面 + 简单逻辑 | ❌ 直接写在页面里 |

### Util 抽离判断

#### 判断依据

1. **纯函数特性**
   - 纯计算/转换/校验
   - 无副作用
   
2. **无 Vue API 依赖**
   - 不依赖 ref/reactive/computed 等
   - 可独立测试

#### 决策规则

| 情况 | 建议 |
|------|------|
| 纯函数 + 无依赖 | ✅ 建议抽离 |
| 有 Vue API 依赖 | → 考虑抽离为 Hook |
| 强业务绑定 | ❌ 不抽离 |

### Store 创建判断

#### 判断依据

1. **全局状态管理需求**
   - 用户信息、主题、全局 loading
   - 需要跨组件共享
   
2. **跨页面共享状态**
   - 多个页面需要访问同一状态
   - 需要持久化存储

#### 决策规则

| 情况 | 建议 |
|------|------|
| 全局状态 + 跨页面 | ✅ 建议创建 Store |
| 单页面状态 | ❌ 使用 ref/reactive |
| 临时状态 | ❌ 不需要 Store |

## 第4步：输出决策清单（⚠️ 必须暂停）

**这是最关键的步骤！AI 必须在此步骤暂停，等待用户确认所有决策。**


- ❌ 跳过此步直接输出定稿提案

- ❌ AI 自行决定资源创建不问用户
- ❌ 把 AI 的建议直接当确定方案写入提案

## 第5步：任务编排

**目的**：根据用户确认的决策，编排任务顺序

### 任务依赖关系

```
create-api（最底层，无依赖）
    ↓
create-store
    ↓
create-hook / create-util
    ↓
create-ui（组件）
    ↓
create-ui（页面）
    ↓
create-logic（最顶层，依赖所有前置资源）
```

### 排序规则

1. **依赖优先**：被依赖的任务排在前面
2. **层级优先**：底层资源优先创建
3. **并行可能**：无依赖的任务可以并行

### 任务编排示例

```markdown
## 任务清单

| 序号 | 任务名称 | 触发 Skill | 依赖 | 状态 |
|------|---------|-----------|------|------|
| T1 | 接入登录 API | create-api | — | ⬜ |
| T2 | 创建用户 Store | create-store | — | ⬜ |
| T3 | 创建密码输入组件 | create-ui | — | ⬜ |
| T4 | 创建登录页 UI | create-ui | T3 | ⬜ |
| T5 | 创建登录页逻辑 | create-logic | T1, T2, T4 | ⬜ |
```

## 第8步：执行任务

### 执行协议

对每个任务，按以下步骤执行：

1. **调用 Skill**：使用 `use_skill("<skill-name>")` 加载对应 Skill
2. **传入需求**：将该任务的"需求描述"作为 Skill 的输入上下文
3. **Skill 自治**：Skill 自行执行其内部流程
4. **标记完成**：任务完成后，回到 Spec 更新该任务状态为 ✅
5. **进入下一个**：继续下一个任务

### 执行规则

- **严格顺序**：按 Spec 中"七、执行顺序"执行，不跳跃
- **一次一个**：一个任务完成后才执行下一个
- **Skill 自主**：不干预 Skill 的内部决策过程
- **状态同步**：每完成一个任务，同步更新 Spec 文件中的任务状态

## 提案文档管理

### 代码作为真相源

**核心原则**：每次生成提案都全新分析代码库，不依赖历史提案文档。

```
增量迭代流程：
用户提供新需求
    ↓
扫描代码库（获取最新状态）
    ↓
分析新增/修改点
    ↓
复用分析 + 抽离分析
    ↓
输出决策清单
    ↓
用户确认
    ↓
更新提案文档
```

### 活跃文档与归档文档

#### 活跃文档
- **位置**：`docs/specs/<name>-spec.md`
- **内容**：只包含当前迭代的任务
- **状态**：待执行任务，保持精简

#### 归档文档
- **位置**：`docs/specs/.archive/<name>/<version>-tasks.md`
- **内容**：已完成的任务详情
- **用途**：历史记录，可追溯

### 归档触发机制

1. **手动归档**：用户主动要求归档
2. **自动归档**：活跃文档中已完成任务超过 5 个

### 归档流程

```
1. 扫描活跃文档，找出所有 ✅ 已完成任务
2. 生成归档文件：docs/specs/.archive/<name>/<version>-tasks.md
3. 从活跃文档中删除已完成任务
4. 更新活跃文档的"变更记录"
```

## 职责边界

### create-proposal 的职责

- ✅ 判断：分析需求，识别资源需求
- ✅ 决策：输出决策清单，等待用户确认
- ✅ 编排：动态组合任务，确定执行顺序
- ❌ 不做：不教下游 Skill 怎么实现

### 下游 Skill 的职责

- create-api：接入 API 接口
- create-store：创建 Store
- create-hook：创建 Hook
- create-util：创建工具函数
- create-ui：创建 UI（模板+样式）
- create-logic：创建逻辑代码

**关键原则**：
- create-proposal 只说"做什么"
- 下游 Skill 根据规范决定"怎么做"
- 下游 Skill 不做抽离判断（已在 create-proposal 中完成）

## 与其他技能的关系

```
design-analysis（眼睛）
       ↓ 产出 UI 清单
create-proposal（大脑）← API 文档 / 交互文档
       ↓ 产出 Spec（任务清单 + 决策记录）
┌──────┼──────┐
↓      ↓      ↓
create-api  create-store  create-hook  create-util  ...（手）
       ↓
   create-ui（UI 实现）
       ↓
   create-logic（逻辑实现）
```

**协作流程**：
1. design-analysis 分析设计稿 → 生成 UI 清单
2. create-proposal 综合分析 → 生成提案
3. 下游 Skill 按提案执行 → 完成开发
