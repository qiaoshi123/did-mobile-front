## 模板使用说明

1. **⛔ 任务详情只写"需求描述"，绝对禁止写实现细节**：说清楚这个任务要实现什么功能。禁止写：代码块、函数签名、TS 接口/类型定义、正则表达式、import 语句、具体字段名/参数名/返回值类型。具体实现由对应的 create-xxx Skill 根据需求和项目规范自行完成。
2. **决策记录是核心章节**：Spec 的长期价值在于决策记录。每个用户确认过的决策都要记录在"决策记录"章节。**包含显式决策（用户主动选择）和隐含决策（分析过程中确认的"不抽离"决策）。**
3. **create-sfc 页面任务引用 UI 清单**：不复制样式数值，写明 UI 清单文件路径，让 create-sfc Skill 自行读取。
4. **资源方案总览合并为一章**：组件、Store、Hook、Util、路由常量的方案合并展示，一目了然。
5. **已有资源标注"直接使用"**：不为已有资源生成任务。
6. **未确认的信息标 ⚠️**：统一用"⚠️ 待补充"标注，不要自行填充。
7. **一个任务 = 一个 Skill 调用**：不同 Skill 范畴的工作必须拆为独立任务。路由常量（`routes.ts`）和路由配置（`pages.json`）属于同一 Skill（`create-route`），应合并为一个任务，不要拆分。
8. **路由配置统一使用 `create-route`**：新建/修改页面路由配置（含路由常量和 pages.json）统一分配给 `create-route` Skill。禁止分配给 `create-util`（路由常量不是工具函数）或 `create-sfc`（pages.json 不是 SFC 文件）。
9. **页面基本信息中的导航栏配置**：有设计稿时从设计稿提取导航栏样式（标题、字体颜色、背景色），无设计稿时导航栏字体颜色和背景色填"继承全局"。这些信息会作为 `create-route` 的输入。
10. **文件路径和命名必须遵循项目 Rule 规范**：产出文件的路径、文件名、目录结构必须严格按照对应 `.mdc` 规则文件中的规定。命名前先扫描代码库中同类资源的现有命名风格。
11. **变更记录是版本追溯的关键**：每次通过变更模式修改 Spec 时，必须在变更记录章节追加新版本。版本号递增（v2、v3...），需包含变更原因、具体变更内容和受影响的任务编号。**创建模式只写 v1，后续版本由变更模式追加。**
12. **变更模式下任务编号不重排**：删除任务后，保留原编号不变，不对后续任务重新编号，以保持跨版本引用的稳定性。新增任务在现有最大编号上递增。
13. **变更模式下决策编号不重排**：同任务编号规则，决策变更时保留原编号更新内容，新决策在最大编号上递增。
