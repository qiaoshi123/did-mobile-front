# 设计稿分析

## 使用时机

当满足以下任一情况时使用本技能：

- 需要**分析设计稿截图**，把界面结构、组件拆分、样式信息梳理成结构化的 UI 清单
- 需要产出一份 **UI 清单文档**，供 `create-proposal` Skill 制定实施计划

## 必须遵循的规范

执行此技能前，**必须先读取以下 Rule 文件**：

- `.codebuddy/rules/04-组件规范.mdc` — 组件分类、选择优先级、命名约定
- `.codebuddy/rules/08-样式规范.mdc` — 变量体系、尺寸单位、布局方式

> 编码规范（TypeScript、命名约定）已作为 always apply rule 自动生效，无需手动读取。

---

## 核心原则

1. **设计稿基准尺寸为 750px 宽**。用户提供的截图尺寸可能不是 750，但所有分析产出的尺寸值必须**换算到 750 基准**（即 `1px 设计值 = 1rpx`）
2. **必须按「从上到下、从左到右、从外到里」的顺序分析**，逐条准确记录。详见 `rules/analysis-order.md`
3. **文字、图片、布局、层级四类信息不允许遗漏**。详见 `rules/analysis-priorities.md`
4. **组件拆分必须对照已有组件库**，遵循组件选择优先级。详见 `rules/workflow-component-mapping.md`
5. **图片/图标只记录位置和尺寸，实现时统一使用占位图**。不需要从截图中提取、切图或下载任何图片资源，编码时统一使用占位写法（如 `placeholder`）
6. **TDesign 变量优先，设计稿色值服从组件库**。当设计稿颜色与 TDesign 变量色值功能定位相同但数值略有差异时（如都是"品牌蓝"但色号不同），一律映射到 TDesign 变量，不追设计稿的精确色值
7. **忽略系统状态栏**。移动端设计稿截图顶部通常带有手机系统状态栏（时间、电池、Wi-Fi 信号等），这部分**不属于页面内容，分析时必须跳过**，不记录到 UI 清单中
8. **多状态截图以基准态为主，差异态只记增量**。用户可能提供同一页面不同状态的多张截图（如表单空态 vs 填写态）。选择一个状态作为基准态做完整分析，其余状态**只记录与基准态的差异**（如文字内容变化、颜色变化、元素显隐、按钮状态切换），最终产出**一份 UI 清单文档**

## 输入物料

| 物料 | 必须 | 说明 |
|------|------|------|
| 设计稿截图 | ✅ | 页面或模块的截图（可以是**多张**，表示同一页面的不同状态，如空态/填写态/错误态） |
| 页面/模块名称 | ✅ | 用于文档命名和组件命名空间 |
| 设计稿实际宽度 | 可选 | 如果截图尺寸不是 750，需告知实际设计宽度（默认 750） |

> **多张截图时**：先确认各截图是"同一页面不同状态"还是"不同页面"。同一页面的多状态截图产出一份 UI 清单；不同页面则分别产出各自的 UI 清单。

## 工作流程（5 步）

### 第 1 步：建立布局 Map

分析截图整体结构，记录页面状态、整体尺寸、区域划分。如果用户提供了**多张截图（多状态）**，在此步骤中识别基准态和差异态。

详见 `rules/workflow-layout-map.md`。

### 第 2 步：区域与元素提取

对每个区域按「从外到里」逐项提取，确保文字、图片、布局、层级四者均准确记录。

详见 `rules/workflow-element-extraction.md`。

### 第 3 步：样式规范汇总

汇总截图中的颜色、字体、圆角、间距等样式信息，**与 TDesign CSS Variables 做映射**。

工作要点：
- 提取到的颜色/间距/圆角，查找是否有对应的 `--td-xxx` 变量
- 有对应变量 → 记录变量名 + 原始值
- 无对应变量 → 记录原始值 + 标注"非 TDesign 变量"
- 所有尺寸值使用 `rpx`（基于 750 基准）

### 第 4 步：组件映射

将分析出的 UI 区块映射到具体的组件方案，遵循组件选择优先级。

详见 `rules/workflow-component-mapping.md`。

### 第 5 步：输出 UI 清单文档

将分析结果输出为标准化文档。

详见 `rules/workflow-output-checklist.md` 和 `rules/output-ui-inventory.md`。

## 尺寸换算规则

用户提供的截图宽度可能不是 750px。必须按以下方式换算：

```
换算公式：目标值(rpx) = 截图测量值 × (750 ÷ 截图实际宽度)

示例：
- 截图宽度 375px，测量到某元素宽 100px → 100 × (750 ÷ 375) = 200rpx
- 截图宽度 750px，测量到某元素宽 100px → 100 × (750 ÷ 750) = 100rpx
- 截图宽度 未知，无法精确测量 → 基于视觉比例估算，并标注"估算值"
```

**注意**：
- 如果用户未提供截图实际宽度，询问用户或根据截图的像素宽度推算
- 所有产出文档中的尺寸值必须是换算后的 rpx 值
- 如果无法精确换算，标注为"≈ xxxrpx（估算）"

## 与其它技能的关系

- **create-proposal**：本技能产出的 UI 清单是 `create-proposal` 的核心输入之一。先跑 `design-analysis` 产出 UI 清单 → 再跑 `create-proposal` 制定实施计划
- **create-page**：开发页面时，引用 UI 清单中的区域描述和样式规范进行精确还原
- **create-component**：开发组件时，引用 UI 清单中的组件映射表确定 Props 和样式

## 检查清单

分析完成、输出文档前，必须逐项检查：

- [ ] 已读取 `04-组件规范` 和 `08-样式规范` Rule
- [ ] 所有尺寸值已换算到 750 基准（rpx）
- [ ] 分析顺序正确（从上到下、从左到右、从外到里）
- [ ] 所有可见文字已记录（内容 + 样式）
- [ ] 所有图片/图标区域已记录（尺寸 + 位置 + 占位方式）
- [ ] 布局参数完整（间距、对齐、排列方向）
- [ ] 层级嵌套关系正确（树形结构图）
- [ ] 颜色/间距/圆角已与 TDesign CSS Variables 做映射
- [ ] 组件映射表已完成（含选择优先级说明）
- [ ] 已扫描 `src/components/index.ts` 确认已有组件
- [ ] 多状态截图已标注基准态，差异态已记录增量变化（如有）
- [ ] 文档路径和命名正确
